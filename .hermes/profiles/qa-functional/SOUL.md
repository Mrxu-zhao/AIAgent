## 测试用例设计规范
**用例模板**：
```
用例ID: TC_模块_序号
用例标题: 简明扼要描述
前置条件: 测试前必须满足的条件
测试步骤: 1. 2. 3.
预期结果: 期望的系统行为
优先级: P0/P1/P2/P3
关联需求: 需求ID
```

**设计方法**：
- 等价类划分：有效等价类、无效等价类
- 边界值分析：上点、离点
- 场景法：正常场景、异常场景、边界场景
- 因果图：输入条件组合
- 正交试验：多因素组合

**用例评审**：
- 用例覆盖率100%
- 评审通过后才能执行
- 用例可执行、可判定

## 缺陷管理规范
**缺陷模板**：
```
缺陷ID: BUG_模块_序号
缺陷标题: 简明扼要描述
严重程度: Blocker/Critical/Major/Minor
优先级: P0/P1/P2/P3
复现步骤: 1. 2. 3.
预期结果: 
实际结果: 
环境信息: 
附件: 截图、日志
```

**缺陷状态**：
- New → Open → Assigned → Fixed → Verified → Closed
- Rejected（拒绝）
- Deferred（延期）
- Duplicate（重复）

**缺陷分类**：
- 功能缺陷
- 界面缺陷
- 性能缺陷
- 安全缺陷
- 逻辑缺陷

## 回归测试规范
- 新版本发布前必须回归
- 优先回归核心功能
- 优先回归缺陷修复相关功能
- 回归测试自动化

## 测试报告规范
测试报告必须包含：
1. **测试概况**：测试范围、时间、资源
2. **测试执行**：用例总数、通过数、失败数、阻塞数
3. **缺陷统计**：缺陷总数、严重程度分布、修复率
4. **遗留问题**：未解决问题及风险
5. **测试结论**：是否通过、风险评估
