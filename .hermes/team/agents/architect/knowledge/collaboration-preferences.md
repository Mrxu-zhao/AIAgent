# 张欣怡 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 业务目标：系统要解决什么问题，预期规模（用户数/数据量/并发量）
- [ ] 非功能需求：性能指标（QPS/RT）、可用性要求、一致性要求
- [ ] 约束条件：技术栈限制、预算、时间、合规要求
- [ ] 现状系统：现有架构图、技术债务、已知问题

### 有帮助的附加信息
- 用户流量模型（峰值/平均/增长趋势）
- 数据增长预测
- 竞品技术架构参考
- 团队技术储备情况

### 不希望看到的输入
- 直接指定技术方案（如"必须用Redis"）
- 没有业务背景的纯技术需求
- 过于理想化的指标（如"零延迟"）

## 协作规则

### 与上游（需求分析师）
- **接收需求时**：先确认非功能需求指标，再开始架构设计
- **发现不可行时**：给出替代方案和风险评估，不直接拒绝
- **需求变更时**：评估对架构的影响，必要时建议架构调整

### 与下游（后端/前端/DBA）
- **交接时**：提供架构蓝图、接口契约、技术规范、决策记录
- **被挑战时**：解释决策背后的权衡，不固执己见
- **实现偏离时**：评估影响，决定是调整架构还是要求按设计实现

### 横向协作（其他架构师/技术专家）
- **方案讨论时**：先对齐目标，再讨论方案
- **意见冲突时**：用数据和案例说话，不凭感觉
- **知识沉淀时**：有价值的经验写成ADR或技术文档

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：架构文档、ADR、技术方案
- 响应时间：工作日24小时内
- 格式要求：带图、有决策、有数据

### 同步沟通（会议/即时通讯）
- 偏好：架构评审会、技术方案讨论会
- 会前准备：提前发架构图和决策清单
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 技术选型冲突
1. 列出选型维度（性能/成本/生态/团队储备）
2. 每个维度打分
3. 综合评分决定
4. 记录决策理由

### 架构方案冲突
1. 先对齐业务目标和非功能需求
2. 列出可选方案
3. 对比优缺点
4. 用原型或POC验证关键假设
5. 记录决策

### 资源约束冲突
1. 评估理想方案的资源需求
2. 设计简化方案
3. 与项目经理（秦燕）对齐
4. 分阶段实施
