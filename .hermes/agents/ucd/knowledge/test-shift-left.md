# 测试左移：需求评审与设计评审参与指南

> 作者：郑晓彤 | 创建时间：2026-04-29 | 适用：测试工程师

## 1. 测试左移的价值

### 传统模式 vs 左移模式

```
传统模式（测试后置）：
需求设计 → 开发编码 → 开发自测 → 测试介入 → 上线
                ↑                        ↑
              这里才发现问题           修复成本高

左移模式（测试前置）：
需求评审 → 设计评审 → 开发编码 → 测试介入 → 上线
     ↑            ↑                    ↑
   早期发现    早期发现            胸有成竹
```

### 测试左移收益

```markdown
1. 降低修复成本
   - 需求阶段修复：1倍成本
   - 设计阶段修复：5倍成本
   - 开发阶段修复：10倍成本
   - 测试阶段修复：20倍成本
   - 上线后修复：100倍成本

2. 提高测试质量
   - 深入理解需求，测试覆盖更全面
   - 提前识别风险，制定应对策略
   - 测试用例更精准，减少无效测试

3. 减少返工
   - 减少因需求理解偏差导致的返工
   - 减少因设计缺陷导致的架构调整
```

## 2. 需求评审参与

### 评审前准备

```markdown
1. 提前获取需求文档
   - PRD（产品需求文档）
   - 业务流程图
   - 原型设计稿
   - 竞品分析（如果有）

2. 预习与思考
   - 理解业务背景和目标
   - 列出可能的疑问点
   - 思考边界情况和异常场景
   - 查阅历史项目经验

3. 准备评审清单
   - 功能完整性
   - 需求清晰度
   - 可测试性
   - 风险点识别
```

### 评审关注点

#### 2.1 功能完整性

```markdown
□ 业务流程是否完整闭环？
  - 开始→过程→结束，每步都有定义？
  - 异常流程有处理？

□ 角色权限是否清晰？
  - 有哪些角色？
  - 每个角色能做什么？
  - 角色间关系？

□ 数据流向是否明确？
  - 数据从哪里来？
  - 怎么处理？
  - 输出什么？

□ 涉及哪些外部系统？
  - 是否有依赖？
  - 异常情况如何处理？
```

#### 2.2 需求清晰度

```markdown
□ 术语定义是否统一？
  - "用户"是指登录用户还是系统用户？
  - "启用"和"激活"是一个意思吗？

□ 边界条件是否明确？
  - 最少/最多支持多少？
  - 最大文件上传大小？
  - 数据保留多长时间？

□ 异常情况是否有定义？
  - 网络断开怎么办？
  - 对方系统挂了怎么办？
  - 数据超限怎么办？

□ 规则是否可量化？
  - "及时"是多久？
  - "优先"是什么规则？
```

#### 2.3 可测试性评估

```markdown
□ 功能是否可以独立测试？
  - 是否强依赖其他功能？
  - 是否需要准备大量前置数据？

□ 验收标准是否明确？
  - 怎样算"完成"？
  - 怎样算"质量合格"？

□ 性能要求是否明确？
  - 响应时间要求？
  - 并发用户数？

□ 数据准备方案？
  - 测试数据从哪里来？
  - 如何隔离测试数据？
```

#### 2.4 评审提问模板

```markdown
需求理解类：
1. "这里的[术语]是指...对吗？"
2. "如果用户...会怎样？"
3. "这个功能和XX功能的区别是什么？"

异常场景类：
1. "如果...会发生什么？"
2. "网络异常/超时怎么处理？"
3. "对方系统不可用时如何处理？"

边界条件类：
1. "最多/最少支持多少？"
2. "超过限制怎么处理？"
3. "从1开始还是从0开始？"

安全权限类：
1. "哪些角色可以操作这个功能？"
2. "未登录用户可以访问吗？"
3. "如何防止越权操作？"

数据相关类：
1. "数据从哪里获取/产生？"
2. "数据保留多长时间？"
3. "删除后数据还能恢复吗？"
```

## 3. 设计评审参与

### 评审前准备

```markdown
1. 获取设计文档
   - 系统架构图
   - 数据库设计（ER图）
   - 接口设计文档
   - 详细设计文档

2. 了解技术栈
   - Spring Boot 版本
   - MyBatis-Plus 用法
   - 数据库选型
   - 缓存策略

3. 准备设计评审清单
```

### 评审关注点

#### 3.1 数据库设计评审

```markdown
□ 表结构是否合理？
  - 字段类型选择
  - 字段长度设置
  - 是否冗余

□ 索引设计是否正确？
  - 查询字段是否建索引
  - 联合索引顺序
  - 是否会产生索引失效

□ 约束是否完整？
  - 主键、外键
  - 唯一约束
  - 非空约束
  - 默认值

□ 关联关系是否清晰？
  - 一对一/一对多/多对多
  - 外键是否级联操作

□ 敏感数据是否加密？
  - 密码
  - 手机号
  - 身份证
```

**示例提问：**

```markdown
1. "用户名有唯一性要求吗？是否需要建唯一索引？"
2. "状态字段有哪些枚举值？后续会新增吗？"
3. "这个表预计数据量多大？是否需要分区？"
4. "删除用户时，相关订单怎么处理？"
5. "密码字段是否加密存储？用什么算法？"
```

#### 3.2 接口设计评审

```markdown
□ 接口设计是否规范？
  - RESTful风格
  - 命名规范
  - HTTP方法正确

□ 请求参数是否合理？
  - 必填/可选区分
  - 参数类型和校验规则
  - 分页参数定义

□ 响应结构是否统一？
  - 成功/失败响应格式
  - 错误码规范
  - 分页数据结构

□ 接口幂等性？
  - POST是否支持重复提交
  - DELETE是否幂等

□ 权限控制策略？
  - 需要登录？
  - 需要什么角色？
```

**示例提问：**

```markdown
1. "这个接口是新增还是更新？是否支持部分更新？"
2. "删除接口是物理删除还是逻辑删除？"
3. "接口调用频率有限制吗？"
4. "是否有批量操作需求？批量大小限制？"
5. "列表接口支持哪些排序字段？"
```

#### 3.3 业务流程评审

```markdown
□ 流程是否完整？
  - 正常流程
  - 异常流程
  - 边界流程

□ 状态流转是否清晰？
  - 状态机定义
  - 哪些操作会触发状态变化
  - 状态变化的前置条件

□ 并发情况处理？
  - 多用户同时操作同一数据
  - 锁机制
  - 乐观锁/悲观锁

□ 事务边界？
  - 哪些操作需要在同一个事务
  - 事务传播行为
```

**示例提问：**

```markdown
1. "审批流程有超时机制吗？超时后如何处理？"
2. "如果两个人同时审批同一个申请，先到先得还是都可以审批？"
3. "状态变更失败时，数据回滚吗？"
4. "长流程中断了，如何恢复或重试？"
```

#### 3.4 性能设计评审

```markdown
□ 性能要求？
  - 响应时间要求
  - 并发用户数
  - QPS/TPS

□ 大数据量处理？
  - 分页查询
  - 异步处理
  - 消息队列

□ 缓存设计？
  - 哪些数据需要缓存
  - 缓存策略（TTL、更新机制）
  - 缓存一致性

□ 索引设计？
  - 查询条件字段
  - 排序字段
  - 分页优化
```

## 5. 评审技巧

### 如何提出有效问题

```markdown
1. 用"如果...会怎样"引导思考
   - "如果用户删除时还有未完成的订单会怎样？"

2. 引用竞品或行业惯例
   - "XX平台的做法是...，我们这边呢？"

3. 提供多个选项
   - "可以采用A方案或B方案，各有优缺点..."

4. 用数据说话
   - "根据XX项目的经验，这类问题会导致..."

5. 先肯定再提建议
   - "整体设计很好，对于XX部分，我有几点建议..."
```

### 如何处理争议

```markdown
1. 先理解对方立场
   - "我理解你的顾虑是..."

2. 聚焦目标和风险
   - "我们的目标是XX，现在的问题是..."

3. 寻求第三方意见
   - "这个问题比较复杂，建议拉上XX一起讨论"

4. 记录分歧
   - 如果无法达成一致，记录分歧点，由PM决策
```

### 需求评审前

```markdown

□ 列出所有疑问点

□ 准备评审问题清单
```

### 需求评审中

```markdown
□ 确认业务流程完整性
□ 确认角色权限定义
□ 确认边界条件和异常处理
□ 确认验收标准
```
