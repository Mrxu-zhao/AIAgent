## 五、测试闭环详细规范

### 5.1 单元测试（★ 开发必须编写）

**前置原则：开发完成代码的同时必须完成单元测试，单元测试未编写不允许提测**

#### 前端单元测试

| 模块 | 测试内容 | 覆盖率要求 | 测试工具 |
|------|----------|-----------|----------|
| API 层 | 接口调用、参数校验、响应处理 | ≥ 70% | Vitest |
| Store 层 | 状态管理、业务逻辑、mutations | ≥ 80% | Vitest + Pinia |
| 组件层 | 渲染、交互、props/emits | ≥ 60% | Vitest + Vue Test Utils |
| 工具层 | 工具函数、格式化、验证 | ≥ 80% | Vitest |

**前端单元测试示例：**
```typescript
// src/api/__tests__/user.test.ts
describe('用户API测试', () => {
  it('登录成功', async () => {
    const res = await login({ username: 'test', password: '123456' })
    expect(res.code).toBe(200)
    expect(res.data.token).toBeDefined()
  })
  
  it('登录失败-密码错误', async () => {
    const res = await login({ username: 'test', password: 'wrong' })
    expect(res.code).toBe(401)
  })
})

// src/stores/__tests__/user.test.ts
describe('用户状态测试', () => {
  it('设置用户信息', () => {
    const store = useUserStore()
    store.setUserInfo({ id: 1, name: '张三' })
    expect(store.userInfo.name).toBe('张三')
  })
})
```

#### 后端单元测试

| 模块 | 测试内容 | 覆盖率要求 | 测试工具 |
|------|----------|-----------|----------|
| Controller | 参数校验、接口响应、异常处理 | ≥ 60% | JUnit5 |
| Service | 业务逻辑、数据处理、边界条件 | ≥ 80% | JUnit5 + Mockito |
| Mapper | SQL执行、数据映射 | ≥ 70% | JUnit5 |
| Utils | 工具方法、算法实现 | ≥ 80% | JUnit5 |

**后端单元测试示例：**
```java
// service/UserServiceTest.java
@SpringBootTest
class UserServiceTest {
    
    @Autowired
    private UserService userService;
    
    @MockBean
    private UserMapper userMapper;
    
    @Test
    void testLogin_Success() {
        when(userMapper.selectOne(...)).thenReturn(user);
        Result<LoginVO> result = userService.login(loginDTO);
        assertEquals(200, result.getCode());
        assertNotNull(result.getData().getToken());
    }
    
    @Test
    void testLogin_PasswordError() {
        when(userMapper.selectOne(...)).thenReturn(null);
        Result<LoginVO> result = userService.login(loginDTO);
        assertEquals(401, result.getCode());
    }
}
```

#### 单元测试交付标准

| 检查项 | 要求 | 检查人 |
|--------|------|--------|
| 测试用例数 | 每个Service方法 ≥ 3个用例 | 开发自检 |
| 覆盖率 | 达到规定覆盖率 | CI检查 |
| 测试通过率 | 100%通过 | CI检查 |
| 测试可维护性 | 命名规范、可读性强 | 代码审查 |

### 5.3 缺陷管理

**缺陷状态流转：**
```
新建 → 分配给开发 → 开发修复中 → 已修复 → 测试验证
                                                    ↓
                                              验证通过 → 关闭
                                              验证失败 → 打回重新修复
```

**缺陷关闭标准：**
- P0（严重）：100%关闭
- P1（高）：100%关闭
- P2（中）：≥ 90%关闭

### 5.4 回归测试

**触发条件：**
- 所有缺陷修复完成后
- 代码变更可能导致功能影响时

**验证内容：**
- 原缺陷是否真正修复
- 修复是否引入新问题
- 核心功能是否正常

### 5.5 测试闭环检查清单

发布前必须全部通过：
- [ ] 前端单元测试覆盖率 ≥ 60%
- [ ] 后端单元测试覆盖率 ≥ 70%
- [ ] 功能测试用例执行率 100%
- [ ] 冒烟测试通过率 100%
- [ ] 所有 P0/P1 缺陷已关闭
- [ ] P2 缺陷关闭率 ≥ 90%
- [ ] 回归测试无新问题引入
