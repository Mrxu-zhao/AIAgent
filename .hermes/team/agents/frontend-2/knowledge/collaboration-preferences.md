# 周晓明 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 数据格式：接口返回的数据结构、字段类型、示例值
- [ ] 交互流程：用户操作流程、状态变化、异常处理
- [ ] 性能要求：渲染时间、内存限制、帧率要求
- [ ] 设计规范：颜色、字体、图表样式、布局规则

### 有帮助的附加信息
- 参考网站或竞品
- 数据量级（条数、更新频率）
- 浏览器兼容性要求
- 网络环境（内网/外网、带宽）

### 不希望看到的输入
- "做个图表"（没有数据格式和样式要求）
- 只有截图没有标注的设计稿
- 没有性能要求的复杂可视化需求

## 协作规则

### 与上游（UCD/后端）
- **接收设计稿时**：先确认交互说明和数据格式
- **数据有问题时**：及时与后端对齐，不擅自修改数据
- **实现有难点时**：提供技术限制和替代方案

### 与下游（测试/运维）
- **交接时**：提供测试数据、性能指标、部署说明
- **Bug修复时**：提供复现步骤和修复方案
- **性能问题**：提供优化前后的对比数据

### 横向协作（其他前端）
- **架构设计时**：与相关方对齐技术方案
- **组件共享时**：提供文档和示例
- **知识沉淀时**：有价值的经验写成文档

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：原型链接、数据格式文档、性能报告
- 响应时间：工作日24小时内
- 格式要求：带图、有数据、可复现

### 同步沟通（会议/即时通讯）
- 偏好：方案评审会、联调会议、性能分析会
- 会前准备：提前发原型或问题列表
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 数据格式冲突
1. 对齐业务需求
2. 评估前后端实现成本
3. 选择性价比最高的方案
4. 记录决策

### 性能优化冲突
1. 定位性能瓶颈
2. 列出优化方案
3. 评估收益和成本
4. 用数据验证

### 技术选型冲突
1. 列出可选方案
2. 评估团队储备
3. 用POC验证
4. 记录决策
