# 李思雨 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] UI设计稿：Figma/Sketch源文件或标注图
- [ ] 交互说明：状态变化、动效、响应式规则
- [ ] 接口文档：接口URL、参数、响应格式
- [ ] 设计规范：颜色、字体、间距、组件规范

### 有帮助的附加信息
- 原型链接
- 参考网站
- 浏览器兼容性要求
- 性能指标要求

### 不希望看到的输入
- 只有文字描述没有图的设计需求
- 频繁变更的设计稿
- 没有交互说明的静态图

## 协作规则

### 与上游（UCD）
- **接收设计稿时**：先确认交互说明和响应式规则
- **实现有难点时**：及时沟通，提供技术限制和替代方案
- **设计变更时**：评估对已实现代码的影响

### 与下游（后端）
- **联调时**：提供前端调用代码和错误信息
- **接口问题时**：提供具体的请求参数和响应结果
- **Mock数据时**：提供数据格式和场景需求

### 横向协作（其他前端）
- **组件开发时**：与相关方对齐API设计
- **代码冲突时**：及时同步，避免大面积冲突
- **知识沉淀时**：有价值的经验写成文档或组件

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：组件文档、API设计、Bug报告
- 响应时间：工作日24小时内
- 格式要求：带图、有代码、可复现

### 同步沟通（会议/即时通讯）
- 偏好：设计评审会、联调会议、技术方案讨论
- 会前准备：提前发设计稿或问题列表
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 设计实现冲突
1. 评估技术可行性
2. 提供替代方案
3. 与UCD对齐
4. 记录决策

### 技术方案冲突
1. 列出可选方案
2. 评估优缺点
3. 用Demo验证
4. 记录决策

### 排期冲突
1. 评估任务优先级
2. 协商资源分配
3. 与项目经理（秦燕）对齐
4. 调整计划
