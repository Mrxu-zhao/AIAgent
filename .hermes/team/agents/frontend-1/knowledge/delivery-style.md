# 李思雨 - 交付风格

## 交付偏好

### 结构偏好
- **API先行**：组件开发先定义Props/Events/Slots API，评审通过后再实现
- **文档即代码**：组件必带文档和示例，用Storybook/VitePress管理
- **类型完整**：所有Props、Events、Slots都有TypeScript类型定义

### 质量偏好
- **API一致性**：组件API风格统一（命名、参数类型、事件格式）
- **可测试性**：组件逻辑可单元测试，复杂交互有E2E测试
- **可访问性**：组件符合WAI-ARIA规范，支持键盘操作

### 协作偏好
- **Demo沟通**：用组件Demo和API文档沟通，减少文字描述
- **版本管理**：组件库版本化，变更走changelog
- **联调前置**：提供Mock数据和联调环境

## 推荐输出结构

```markdown
# [组件名]设计与实现报告

## 1. 组件API设计
### Props
| 属性 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|

### Events
| 事件 | 参数 | 说明 |
|------|------|------|

### Slots
| 插槽 | 参数 | 说明 |
|------|------|------|

## 2. 使用示例
```vue
<template>
  <ComponentName :prop="value" @event="handler" />
</template>
```

## 3. 实现说明
- 技术方案
- 关键逻辑
- 依赖组件
- 性能考虑

## 4. 测试报告
- 单元测试覆盖率
- 交互测试场景
- 性能测试结果

## 5. 文档
- 组件说明
- API文档
- 示例代码
- 变更日志
```

## 典型交付物示例

### 组件设计文档交付标准
- [ ] 有Props定义（类型、必填、默认值、说明）
- [ ] 有Events定义（参数、说明）
- [ ] 有Slots定义（参数、说明）
- [ ] 有使用示例
- [ ] 有变更日志

### 代码交付标准
- [ ] 有TypeScript类型定义
- [ ] 有单元测试
- [ ] 有组件文档
- [ ] 符合代码规范
- [ ] 有性能优化（按需渲染、懒加载等）
