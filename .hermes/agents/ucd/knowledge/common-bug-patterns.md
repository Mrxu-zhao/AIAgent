# 常见 Bug 模式总结

> 作者：郑晓彤 | 创建时间：2026-04-29 | 适用：B端管理后台测试

## 1. 空指针异常（NullPointerException）

### 常见场景

```markdown
1. 对象未初始化直接使用
   - 后端：未对查询结果进行null判断
   - 前端：API返回null时直接访问属性

2. 集合操作不当
   - 集合为null时调用size()或遍历
   - 对null集合进行add/remove操作

3. 字符串操作
   - 对null字符串调用trim()、length()
   - null + "字符串" 导致空指针

4. 类型转换
   - 强制转换类型不匹配的对象
   - JSON.parse(null) 返回null未处理
```

### 典型案例

```java
// 案例1：未校验查询结果
User user = userMapper.selectById(id);
user.setName("新名字");  // 如果user为null，这里直接NPE

// 正确写法：
User user = userMapper.selectById(id);
if (user != null) {
    user.setName("新名字");
}

// 案例2：Stream操作未过滤null
List<String> names = users.stream()
    .map(User::getName)
    .collect(Collectors.toList());
// 如果getName()返回null，names列表中会包含null

// 正确写法：
List<String> names = users.stream()
    .map(User::getName)
    .filter(Objects::nonNull)
    .collect(Collectors.toList());
```

### 前端常见空指针

```javascript
// 案例1：访问可能为null的属性
const name = response.data.user.name;  
// 如果response.data.user为null，直接报错

// 正确写法：
const name = response?.data?.user?.name || '';

// 案例2：数组元素可能为空
const firstUser = users[0];
firstUser.name = 'xxx';  // users可能为空数组

// 正确写法：
if (users && users.length > 0) {
    users[0].name = 'xxx';
}
```

### 测试检查点

```markdown
□ 新增记录后立即查询，验证返回完整对象
□ 修改字段后验证非空字段不受影响
□ 删除操作后验证关联查询不会报错
□ 网络异常时验证前端不会崩溃
□ 并发操作时验证返回数据完整性
```

## 2. 边界值相关 Bug

### 数值边界

```markdown
1. 整数溢出
   - 金额字段超过最大值（INT_MAX=2147483647）
   - 计数器超过上限

2. 浮点数精度
   - 0.1 + 0.2 ≠ 0.3
   - 金额计算出现 0.0000001 的误差

3. 分页边界
   - 第0页、超大页码
   - 总数为负数
   - 分页大小为0或负数
```

### 测试用例

```markdown
用例：金额计算精度测试
步骤：
  1. 订单1金额：0.1
  2. 订单2金额：0.2
  3. 订单3金额：0.3
  4. 计算总金额
预期：
  - 结果 = 0.6
  - 不是 0.6000000000000001

用例：超长数据测试
步骤：
  1. 输入用户名：500个字符
预期：
  - 提示"用户名不能超过20个字符"
  - 不应出现截断或报错
```

### 日期边界

```markdown
1. 跨年边界
   - 12月31日 → 1月1日
   - 年度统计

2. 闰年边界
   - 2月29日在非闰年
   - 2月28日/29日的逻辑判断

3. 时区边界
   - UTC vs 本地时间
   - 夏令时切换
```

## 3. 并发相关 Bug

### 常见场景

```markdown
1. 库存超卖
   多个用户同时抢购有限库存

2. 重复提交
   - 双击按钮导致重复创建
   - 网络重试导致重复请求

3. 数据覆盖
   - 两人同时编辑同一记录
   - 后提交的覆盖先提交的

4. 缓存穿透
   - 缓存失效瞬间大量请求打到数据库

5. 分布式锁问题
   - 锁粒度过大导致性能问题
   - 锁未释放导致死锁
```

### 典型案例

```markdown
案例：库存超卖
步骤：
  1. 商品A库存 = 1
  2. 用户1和用户2同时下单
  3. 两人同时查询库存，都看到=1
  4. 两人都下单成功
结果：
  - 库存变成-1
  - 一人付款后无法发货

解决方案：
  - 数据库加唯一索引
  - 使用乐观锁/悲观锁
  - Redis原子操作扣减

案例：重复提交
步骤：
  1. 用户点击"提交订单"按钮两次
  2. 网络较慢，第一请求未完成
  3. 第二请求也发送出去
结果：
  - 创建了两条相同的订单

解决方案：
  - 前端：按钮点击后禁用
  - 后端：请求幂等性处理
  - 使用Token机制防重复
```

### 测试方法

```markdown
并发测试策略：

1. JMeter并发测试
   - 50个线程同时请求
   - 验证库存不会超卖

2. Postman批量运行
   - 使用Collection Runner
   - 迭代次数=10，并发执行

3. 数据库直接验证
   - 并发操作后检查数据一致性
   - 验证事务隔离级别正确
```

## 4. 兼容性 Bug

### 浏览器兼容性

```markdown
1. CSS兼容性问题
   - flex布局在旧浏览器不支持
   - 滚动条样式差异

2. JavaScript兼容性问题
   - ES6+语法在旧浏览器不兼容
   - API差异（fetch vs XMLHttpRequest）

3. Cookie/Session问题
   - SameSite Cookie策略
   - 第三方Cookie被阻止

常见问题浏览器：
  - IE11（正在淘汰但仍有用户）
  - 老版本Chrome/Safari/Firefox
  - 微信内置浏览器（X5内核）
```

### 后端兼容性

```markdown
1. 数据库兼容
   - MySQL 5.7 vs 8.0 语法差异
   - 排序规则（utf8mb4_unicode_ci）

2. JDK版本兼容性
   - JDK8 vs JDK11+ 语法差异
   - 反射/注解行为变化

3. 第三方库版本
   - Spring Boot版本升级
   - MyBatis-Plus版本差异
```

### 测试建议

```markdown
□ 覆盖主流浏览器（Chrome、Firefox、Safari、Edge）
□ 覆盖主流分辨率（1920x1080、1366x768、1440x900）
□ 测试移动端视图（手机、平板）
□ 测试不同操作系统（Windows、macOS、Linux）
□ 测试高DPI屏幕
□ 测试打印功能（CSS @media print）
```

## 5. 权限相关 Bug

### 常见场景

```markdown
1. 越权访问
   - 普通用户访问管理员URL
   - 查看/操作其他用户数据

2. 水平越权
   - 用户A访问用户B的数据
   - 同一角色用户间数据未隔离

3. 垂直越权
   - 低权限用户获得高权限
   - 绕过权限验证

4. 页面权限 vs 接口权限
   - 页面隐藏了按钮但接口未鉴权
   - 直接调用接口可以操作
```

### 典型案例

```markdown
案例：水平越权
步骤：
  1. 用户A登录，获取token
  2. 修改URL中的userId为用户B的ID
  3. 访问 /api/users/2
预期（有Bug）：
  - 返回用户B的详细信息
  - 用户A越权看到了B的数据

正确做法：
  - 后端校验：当前用户只能查看自己的数据
  - 或：管理员可以查看所有人，普通用户只能查看自己

案例：接口权限绕过
步骤：
  1. 页面隐藏了"删除"按钮（无权限）
  2. 用户通过Postman直接调用 DELETE /api/users/1
预期（有Bug）：
  - 操作成功
  - 绕过了前端权限控制

正确做法：
  - 所有接口都需要权限校验
  - 接口返回403而非隐藏按钮
```

### 测试检查点

```markdown
□ 未登录访问需认证接口 → 返回401
□ 普通用户访问管理员接口 → 返回403
□ 用户A访问用户B的数据 → 返回403或无数据
□ 直接URL访问绕过前端权限 → 接口正确拒绝
□ 权限变更后立即生效
□ 子账号权限正确继承
□ Session超时验证
```

## 6. 数据一致性 Bug

### 常见场景

```markdown
1. 缓存与数据库不一致
   - 更新数据库后未更新缓存
   - 缓存过期时间设置不当

2. 主从数据库延迟
   - 写操作后立即读，从库数据未同步

3. 分布式事务问题
   - 部分操作成功，部分失败
   - 分布式锁超时导致数据不一致

4. 批量操作部分失败
   - 批量更新10条，失败2条
   - 没有事务回滚
```

### 典型案例

```markdown
案例：缓存双写不一致
步骤：
  1. 更新用户姓名
  2. 立即查询用户信息
结果（有Bug）：
  - 数据库已更新
  - 缓存返回旧数据
  - 页面显示旧姓名

正确做法：
  - Cache Aside：更新DB后删除缓存
  - Read Through：更新DB后更新缓存
  - 设置合理的缓存过期时间

案例：批量导入部分成功
步骤：
  1. 导入Excel，100条数据
  2. 第50条数据格式错误
  3. 前50条成功，后50条失败
预期（有Bug）：
  - 部分成功，部分失败
  - 没有事务保护

正确做法：
  - 全部成功或全部失败
  - 或明确告知成功数量和失败原因
```

## 7. 性能相关 Bug

### 常见场景

```markdown
1. N+1查询问题
   - 循环中查询数据库
   - 一条SQL变成N条SQL

2. 大数据量处理
   - 一次性查询10万条数据
   - 导出Excel内存溢出

3. 慢查询
   - 未建索引
   - SQL语句低效

4. 前端渲染性能
   - 大列表未分页
   - 频繁DOM操作
```

### 测试方法

```markdown
性能测试关注点：

□ 接口响应时间（<2秒）
□ 页面加载时间（首屏<3秒）
□ 大数据量查询（1000+条）
□ 频繁操作响应（<500ms）
□ 并发用户数支持
□ 内存泄漏检测
```

## 8. Bug 优先级分类

### P0 - 严重

```markdown
□ 数据丢失
□ 系统崩溃
□ 安全性漏洞（数据泄露、SQL注入）
□ 核心业务流程阻塞
□ 资金相关错误
```

### P1 - 高

```markdown
□ 功能缺失
□ 严重性能问题
□ 权限绕过
□ 数据一致性问题
□ 边界条件处理错误
```

### P2 - 中

```markdown
□ 非核心功能错误
□ UI显示问题
□ 提示信息不友好
□ 兼容性问题
□ 偶发性问题
```

### P3 - 低

```markdown
□ UI细节问题
□ 优化建议
□ 文案错误
□ 非关键路径偶发问题
```

## 9. Bug 复现技巧

```markdown
1. 精确记录操作步骤
   - 每一步都记录
   - 包括具体参数值

2. 提供完整环境信息
   - 浏览器版本
   - 操作系统
   - 网络环境

3. 提供完整日志
   - 前端Console日志
   - 后端应用日志
   - 网络请求详情

4. 提供截图/录屏
   - 问题页面截图
   - 操作过程录屏

5. 提供复现频率
   - 必现/偶现（概率多少）
   - 首次发现时间
```
