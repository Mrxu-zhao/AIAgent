## 2. 慢接口定位方法

### 2.1 定位流程

```
慢接口定位四步法:

Step 1: 从报告中找出慢接口
        │
        ↓
Step 2: 确认是普遍慢还是偶发慢
        │
        ↓
Step 3: 分析时间分布(前端/后端/DB)
        │
        ↓
Step 4: 定位具体原因
```

### 2.2 慢接口识别标准

```yaml
慢接口判断标准:

绝对标准:
├── GET请求: RT > 500ms
├── POST请求: RT > 1000ms
├── 复杂查询: RT > 2000ms
└── 文件操作: RT > 5000ms

相对标准 (环比):
├── 相比上次测试: RT增长 > 50%
├── 相比基线: RT增长 > 100%
└── 相比同类接口: RT > 3倍

业务标准:
├── 影响核心业务流程
├── 用户可感知延迟
└── 超时率 > 1%
```

```yaml
响应时间分段统计:

┌─────────────────┬────────┬────────┐
│ 响应时间区间    │ 请求数  │ 占比   │
```

## 3. 瓶颈分析方法

### 3.1 瓶颈分类

```
性能瓶颈类型:

┌────────────────────────────────────────────────────┐
│                      CPU瓶颈                        │
│         现象: CPU高, 但吞吐量低                      │
│         原因: 代码效率低, 死循环, 复杂计算            │
├────────────────────────────────────────────────────┤
│                      内存瓶颈                        │
│         现象: 内存使用率高, Full GC频繁              │
│         原因: 内存泄漏, 大对象, 缓存过大              │
├────────────────────────────────────────────────────┤
│                      IO瓶颈                         │
│         现象: 磁盘IO高, 网络带宽满                    │
│         原因: 日志过多, 大文件传输, 同步阻塞          │
├────────────────────────────────────────────────────┤
│                      连接池瓶颈                      │
│         现象: 连接等待, 超时增多                      │
│         原因: 慢SQL, 连接泄漏, 池太小                │
├────────────────────────────────────────────────────┤
│                      线程池瓶颈                      │
│         现象: 线程耗尽, 请求排队                      │
│         原因: 阻塞操作, 线程泄漏, 池太小              │
└────────────────────────────────────────────────────┘
```

## 4. 典型瓶颈案例

### 4.1 案例1: 数据库连接池耗尽

```
问题现象:
- TPS突然下降
- 响应时间急剧上升
- 日志中出现连接等待

排查步骤:
1. 查看DB连接数
   show status like 'Threads_connected';
   
2. 查看当前连接
   show processlist;
   
3. 查看应用连接池配置
   spring.datasource.hikari.maximum-pool-size=20
   
4. 分析慢SQL
   select * from performance_schema.events_statements_summary_by_digest
   where AVG_TIMER_WAIT > 1000000000;

原因: 连接池太小(20)，但有慢查询占用连接

解决:
- 优化慢查询
- 增大连接池: 20 → 50
- 添加连接超时配置
```

### 4.2 案例2: JVM Full GC 频繁

```
问题现象:
- 内存使用率持续上升
- Full GC频率 > 3次/小时
- 响应时间周期性波动

排查步骤:
1. 查看GC日志
   -XX:+PrintGCDetails -Xloggc:gc.log

2. 分析GC原因
   jstat -gcutil <pid> 1000
   
   S0     S1     E      O      M     YGC     YGCT    FGC    FGCT     GCT
   0.00  65.42  45.23  89.12  78.56  1234   45.67  567    123.45  169.12
   
   O区使用率89%，Old区满了导致频繁Full GC

3. 排查内存泄漏
   jmap -histo <pid> | head -30
   查看内存占用最大的对象

原因: 代码中存在内存泄漏或缓存无限制增长

解决:
- 使用WeakHashMap管理缓存
- 限制缓存大小
- 添加内存监控报警
```

### 4.3 案例3: CPU 100% 持续高

```
问题现象:
- CPU使用率100%
- 系统负载高
- TPS下降

排查步骤:
1. 定位CPU高进程
   top -c
   
2. 定位CPU高线程
   top -Hp <pid>
   
3. 查看线程堆栈
   jstack <pid> > thread.log
   printf "%x\n" <high-pid>  # 转换为16进制
   grep <hex-pid> thread.log

4. 分析热点代码
   使用 arthas 的 profiler 命令
   
原因: 某个接口存在死循环或复杂计算

解决:
- 修复代码死循环
- 优化算法复杂度
- 添加计算缓存
```

## 5. 性能趋势分析

### 5.1 TPS趋势分析

```yaml
TPS变化趋势图:

时间    │  10:00 │  10:15 │  10:30 │  10:45 │  11:00 │  11:15 │
───────┼────────┼────────┼────────┼────────┼────────┼────────┤
TPS    │  880   │  890   │  920   │  850   │  820   │  800   │
RT(ms) │  115   │  112   │  108   │  118   │  122   │  125   │
CPU%   │   65   │   68   │   70   │   72   │   73   │   75   │

分析:
1. TPS在10:30达到峰值920
2. 之后TPS下降，CPU上升 → 出现资源争抢
3. RT随时间缓慢增长 → 可能存在内存泄漏或连接泄漏
4. 建议: 排查10:30后的性能下降原因
```

### 5.2 响应时间百分位分析

```yaml
响应时间分布 (P50/P90/P95/P99):

接口       │ P50   │ P90   │ P95   │ P99   │ Max
───────────┼───────┼───────┼───────┼───────┼──────
/login     │  85ms │ 150ms │ 180ms │ 250ms │ 500ms
/search    │ 200ms │ 400ms │ 500ms │ 800ms │2000ms
/order     │ 350ms │ 600ms │ 800ms │1200ms │3000ms
/redirect  │  50ms │  80ms │ 100ms │ 150ms │ 300ms

分析:
1. /order 的P99=1200ms，超出SLA要求(1000ms)
2. /search 存在长尾问题，Max=2000ms
3. P95与P99差距大，存在极端慢请求
4. 需要重点优化/order和/search接口
```

## 6. 性能问题诊断清单

### 6.1 快速诊断清单

```
性能问题快速诊断:

□ 1. 确认问题范围
  ├─ 全局慢还是单接口慢?
  ├─ 规律性慢还是偶发慢?
  └─ 从什么时候开始变慢?

□ 2. 检查系统资源
  ├─ CPU使用率是否高?
  ├─ 内存是否够用?
  ├─ 磁盘IO是否正常?
  └─ 网络是否拥塞?

□ 3. 检查应用状态
  ├─ JVM堆内存使用率?
  ├─ GC频率是否正常?
  ├─ 线程池是否耗尽?
  └─ 连接池是否耗尽?

□ 4. 检查数据库
  ├─ 是否有慢查询?
  ├─ 连接数是否正常?
  ├─ 索引是否生效?
  └─ 锁等待情况?

□ 5. 检查中间件
  ├─ Redis是否正常?
  ├─ MQ是否积压?
  └─ 缓存是否命中?

□ 6. 查看日志
  ├─ 应用日志是否有异常?
  ├─ GC日志是否有问题?
  └─ 慢查询日志有多少?
```

### 6.2 性能优化优先级

```
优化优先级排序:

P0 - 严重 (立即处理):
├── 系统不可用
├── 致命错误
└── TPS为0

P1 - 高优先级:
├── CPU持续100%
├── 内存泄漏
├── 响应时间 > 5秒
└── 错误率 > 5%

P2 - 中优先级:
├── 响应时间超标
├── P99超标
├── 资源使用率 > 80%
└── 偶发性超时

P3 - 低优先级:
├── 单接口轻微慢
├── 日志优化
└── 代码重构
```
