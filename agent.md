# AI Agent 行为约束规范 v5.0

> 本规范整合仓库级 Agent 工作流、Plan/Spec 强制流程与最小精准修改原则，适用于所有代码分析、审查、修改任务。
> 所有代码修改必须遵循：Plan → Spec → 确认 → Execute 的完整流程。

---

## 核心范式

**基于完整上下文的最小精准修改**：在充分理解代码仓库结构、依赖关系、编码规范的基础上，直接给出唯一必要的修改，不做任何多余动作。

**仓库事实优先**：从仓库实际代码出发，不从通用框架记忆或假设出发。

**零多余动作**：不格式化、不重构、不优化、不添加防御性代码。

**强制 Plan/Spec 流程**：任何代码修改必须先输出 Plan，再输出 Spec，经确认后方可执行。

---

## 强制工作流

### Phase 0: 上下文发现（Discovery）

**修改前必须完成（除非用户明确要求跳过）**

1. **读取仓库入口文档**
   - `README.md`
   - `CODE_WIKI.md`（如存在）
   - 本文件（`agent.md`）

2. **识别代码定位**
   - 当前主实现区：`.hermes/team/control_plane/`
   - 历史兼容区：`.hermes/team/调度框架/`
   - 团队共享知识：`.hermes/team/knowledge/`
   - 角色模板：`.hermes/agents/`
   - 运行时配置：`.hermes/profiles/`
   - 全局行为定义：`.hermes/SOUL.md`

3. **读取相关具体文件**
   - 不停止在目录名或摘要层面
   - 不将单个文件当作整个子系统的代表
   - 区分主实现代码与兼容代码

4. **确认事实后再进入 Plan**
   - `已读文件`：列出实际检查的文件
   - `确认事实`：仅包含这些文件支持的事实
   - `未确认项`：信息缺失时明确说明

> **决策规则**：当不确定时，放慢速度，多读代码。准确性比速度重要，证据比优雅重要。

### Phase 3: Confirm（确认）

**触发条件**：Spec 完成后，必须等待用户明确确认。**禁止在确认前执行任何修改。**

**确认话术模板**：
```
以上 Plan 和 Spec 已准备完毕。

请确认以下事项：
1. 修改范围是否正确？（是/否/需调整）
2. 是否有遗漏的约束？
3. 是否可以开始执行？

确认后，我将严格按照 Spec 进行修改，不会超出范围。
```

**用户确认方式**：
- 明确回复"确认"、"开始"、"执行"
- 或回复"同意"、"OK"、"可以"
- 模糊回复（如"先这样"、"看看"）**不算确认**，需再次确认

---

### Phase 4: Execute（执行）

**触发条件**：收到用户明确确认后，方可执行。**严格执行 Spec，禁止偏离。**

**执行检查清单（每修改一个文件前检查）**：

- [ ] 此文件在 Spec 的修改清单中？
- [ ] 修改行号在 Spec 约定的范围内？
- [ ] 修改类型与 Spec 一致？
- [ ] 没有引入新的 import/依赖？
- [ ] 没有修改接口签名？
- [ ] 没有新增类/方法？（除非 Spec 明确允许）
- [ ] 代码风格与周围代码一致？

## 上下文感知原则

### 必须自动分析的内容（无需用户提示）

1. **项目结构**
   - 自动识别技术栈（Spring/Django/React/Vue 等）
   - 自动识别模块边界和分层架构
   - 自动识别配置文件和依赖管理方式

2. **编码规范**
   - 从现有代码中提取命名规范（camelCase/snake_case/kebab-case）
   - 从现有代码中提取缩进、换行、括号风格
   - 从现有代码中提取注释规范和文档风格
   - 从现有代码中提取错误处理模式

3. **依赖关系**
   - 识别当前文件引用了哪些内部模块
   - 识别当前文件使用了哪些外部库
   - 识别接口实现关系和继承链

4. **代码风格一致性**
   - 与周围代码保持完全一致的写法
   - 使用项目中已有的工具方法和辅助函数
   - 遵循项目中已有的设计模式（如存在）

## 绝对禁止清单（红线）

### 格式与风格
- ❌ **禁止格式化未修改的代码**：即使相邻代码格式混乱，也不触碰
- ❌ **禁止统一代码风格**：不将现有代码改为"更规范"的写法
- ❌ **禁止修改注释**：不增删改任何现有注释（包括拼写错误）
- ❌ **禁止修改空行**：不调整代码间的空行数量

### 架构与结构
- ❌ **禁止重构**：不提取方法、不内联方法、不重命名变量
- ❌ **禁止引入新抽象**：不新增接口、抽象类、基类
- ❌ **禁止迁移模式**：不将现有代码改为其他设计模式
- ❌ **禁止修改配置文件**：不动 pom.xml/build.gradle/package.json 等
- ❌ **禁止新增依赖**：不使用项目中未引入的库

### 行为与逻辑
- ❌ **禁止主动优化**：不优化性能、内存、算法复杂度
- ❌ **禁止添加防御性代码**：不在未明确要求时添加参数校验
- ❌ **禁止修改日志**：不增删改日志输出
- ❌ **禁止修改异常处理**：不改变现有异常类型和处理方式
- ❌ **禁止修改返回值语义**：不改变方法对外暴露的行为契约

### 范围控制
- ❌ **禁止跨文件修改**：除非需求明确涉及多文件
- ❌ **禁止修改接口签名**：不增删参数、不改返回类型、不改抛异常声明
- ❌ **禁止修改测试文件**：除非修复的 bug 有对应测试需要同步

### 工作流红线
- ❌ **禁止在读取相关代码前给出最终方案**
- ❌ **禁止仅从先前经验推断仓库架构**
- ❌ **禁止将单个文件当作整个子系统的代表**
- ❌ **禁止忽视主实现代码与兼容代码的区别**
- ❌ **禁止在解决限定任务时进行无关重构**
- ❌ **禁止覆盖或回退非自己创建的修改**
- ❌ **禁止未经用户明确要求创建新分支**（在 master 上工作）
- ❌ **禁止跳过 Plan/Spec 直接修改代码**
- ❌ **禁止在 Spec 未经确认前执行修改**
- ❌ **禁止执行时偏离 Spec 范围**

## 允许行为清单（白名单）

### 直接允许（无需说明）
- ✅ 修复明确的 bug（空指针、数组越界、逻辑错误等）
- ✅ 实现已明确描述的功能需求
- ✅ 添加明确的边界条件处理
- ✅ 修改明确的配置值（如常量、阈值）

### 条件允许（需在 Plan/Spec 中说明理由）
- ✅ 新增私有方法：当现有方法超过 50 行且逻辑可清晰拆分时
- ✅ 新增局部变量：当表达式嵌套超过 3 层时
- ✅ 修改方法可见性：从 private 改为 package-private（用于测试）

### 必须包含的上下文证明

每次修改必须证明"为什么这是最小区间"：

1. **向上追溯**：说明为什么从这一行开始修改（而非更早）
2. **向下追溯**：说明为什么到这一行结束修改（而非更晚）
3. **横向证明**：说明为什么没有修改相邻的相关代码

## 精准定位规则

### 行号定位精度
- **Bug 修复**：精确到引入 bug 的最小表达式
- **功能实现**：精确到需要插入新逻辑的最小代码块
- **边界处理**：精确到需要添加判断的最小作用域

### 反例纠正

❌ **错误**：修复空指针，顺手把方法拆成了 3 个
❌ **错误**：修复空指针，把相邻的魔法数字改成了常量
❌ **错误**：修复空指针，给方法加了参数校验注解

✅ **正确**：只在取值处添加 `!= null` 判断，其余不动

---

## 变更规则

- 优先最小化、针对性的修改
- 保持现有命名和目录约定，除非任务要求改变
- 添加文档时，关联到真实的仓库路径和工作流预期
- 改变行为时，如果附近存在测试，添加或更新针对性测试

---

## 验证规则

修改后必须：

1. 运行与修改区域最相关的测试或检查
2. 清晰报告结果
3. 如果无法运行验证步骤，明确说明并解释原因

> **绝不**在未提及验证的情况下声称修改已完成。

---

## 审查标准

如果任务是审查：

- 以发现项开头
- 优先关注 bug、回归、错误假设和缺失测试
- 保持摘要简短
- 为每个重要主张引用文件

---

## 特殊情况处理

### 当执行时发现 Spec 不足
**禁止自行补充修改**，必须：
1. 暂停执行
2. 说明发现的问题
3. 提出补充 Plan
4. 等待用户确认新的 Spec
5. 按新 Spec 继续执行

### 当现有架构确实无法支持需求时
1. 在 Plan 阶段说明现有架构的限制
2. 提供最小化的架构调整方案作为备选
3. 明确告知需要新增的接口/模块
4. **等待用户明确选择后再进入 Spec**

### 当发现现有代码有严重设计问题时
1. 记录问题但不主动修改
2. 在 Plan 中作为风险说明
3. 提供修复建议作为独立选项
4. **不将修复混入当前需求中**

## 与 GPT-5 / Claude 的对标要点

| 维度 | GPT-5 / Claude 行为 | 本规范要求 |
|------|-------------------|-----------|
| **上下文理解** | 自动分析整个代码仓库结构 | Discovery 强制工作流 |
| **修改精准度** | 只修改必要的一行/几行 | 精确定位最小修改区间 |
| **风格一致性** | 与现有代码 100% 一致 | 提取并遵循现有编码规范 |
| **零多余动作** | 不格式化、不重构、不优化 | 绝对禁止清单严格执行 |
| **输出形式** | 直接给出 diff，附带简要说明 | Plan + Spec + diff + 上下文依据 |
| **用户交互** | 一次性输出，用户审查即可 | Plan → Spec → 确认 → Execute |

## 示例：完整工作流演示

### 场景：修复空指针异常

**用户输入**：
```java
// UserService.java
public String getUserName(Long userId) {
    User user = userDao.findById(userId);
    return user.getName();  // 这里可能 NPE
}
```

**❌ 错误输出（跳过流程）**：
```java
public String getUserName(Long userId) {
    if (userId == null) {
        throw new IllegalArgumentException("userId cannot be null");
    }
    User user = userDao.findById(userId);
    if (user == null) {
        log.warn("User not found: {}", userId);
        return null;
    }
    return user.getName();
}
```
> 问题：跳过 Plan/Spec，直接修改，且过度修改

**✅ 正确输出（遵循完整流程）**：

---

**Phase 1: Discovery**

```markdown
## 已读文件
- `UserService.java`
- `UserDao.java`

## 确认事实
- `userDao.findById()` 在记录不存在时返回 null
- 项目中无参数校验惯例
- 项目已有全局异常处理

## 未确认项
- 无
```

**Phase 2: Plan**

```markdown
# Plan: 修复 UserService.getUserName 空指针异常

> **Goal:** 修复 `UserService.getUserName` 方法的空指针异常
> **Architecture:** 业务层 Service
> **Tech Stack:** Spring Boot

---

## 现状分析

- **问题定位**：`UserService.java` 的 `getUserName()` 方法，L4
- **影响范围**：仅 UserService 类
- **现有逻辑**：直接从 DAO 获取 User 对象后调用 getName()，未判断 null

## 修改方案

- **方案概述**：在返回值处添加 null 判断
- **修改文件清单**：
  1. `UserService.java` - L4 添加 null 判断
- **新增内容**：无
- **删除内容**：无

## 风险评估

- **接口变更**：否
- **行为变更**：否（原异常场景现返回 null，符合预期）
- **回归风险**：低
```

---

**Phase 3: Spec**

```markdown
# Spec: 修复 UserService.getUserName 空指针异常

> **基于 Plan:** Plan: 修复 UserService.getUserName 空指针异常
> **日期**: 2026-05-19

---

## 1. 背景

`UserService.getUserName` 在 `userDao.findById` 返回 null 时抛出 NPE。

## 2. 目标与非目标

### 2.1 目标

- 修复 NPE，使方法在记录不存在时安全返回 null

### 2.2 非目标

- 不添加参数校验
- 不添加日志
- 不修改 DAO 层

## 3. 详细设计

### 3.1 修改清单

| 序号 | 文件路径 | 修改类型 | 行号范围 | 具体修改 |
|-----|---------|---------|---------|---------|
| 1 | `UserService.java` | 修改 | L4 | 添加 null 判断 |
```
