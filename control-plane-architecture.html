仓库级控制平面总体设计草图

目标不是让现有调度框架立即承担所有职责，而是在全仓库之上增加一个统一的任务编排、状态同步、冲突检测、 并行执行和结果聚合层。调度框架优化只是其中一个并行工作流。
控制平面 并行子任务 / Workstream 状态与协议 汇总与回归验证 故障隔离与回滚

1. 仓库级控制入口

单一总控入口负责读取评估报告、生成任务清单、注册子任务、分配给多个智能体并跟踪生命周期。 它只做编排与治理，不直接承载全部业务逻辑。

2. 统一状态与通信层

任务协议

每个子任务必须有唯一 ID、版本、输入摘要、输出契约、依赖、锁域、验收标准。

状态仓

统一记录 `planned / ready / running / blocked / done / failed / rolled_back` 等状态。

冲突检测

基于文件锁域、模块域、协议版本与结果校验检测冲突，禁止高风险并发写入同一区域。

3. 并行 Workstream 层

WS-A

调度框架正确性修复：路由、工作流、监控死锁、恢复闭环。

WS-B

通信与状态治理：消息契约、共享状态、执行日志、聚合规范。

WS-C

性能与基准：延迟、CPU、吞吐基线，压测脚本，前后对比。

WS-D

文档与可维护性：设计、运行手册、验收标准、变更记录、后续治理手册。

4. 聚合与回归验证层

所有子任务完成后，由统一聚合器汇总结果、执行端到端回归测试并输出性能对比报告。

  • 聚合子任务输出与执行日志
  • 执行全链路回归测试
  • 验证延迟降低 ≥ 20%
  • 验证 CPU 利用率下降 ≥ 15%
  • 生成最终交付文档

5. 故障隔离与回滚层

单个子任务失败时，不阻断其他独立 workstream；仅阻断依赖它的后续任务，并触发局部回滚。

  • 超时自动标记失败
  • 按策略执行有限重试
  • 失败任务生成回滚包与问题快照
  • 隔离对非依赖任务的影响

建议的执行主链路

阶段 1：任务建模 从评估报告提取 P0 / P1 / P2 建议，转换为独立子任务与并行 workstream。
阶段 2：协议建仓 建立任务清单、状态文件、锁域定义、执行日志、结果契约与冲突检测规则。
阶段 3：多智能体并发执行 每个智能体只处理自己负责的子任务，按统一协议上报中间结果。
阶段 4：统一聚合与端到端验证 自动汇总、运行回归测试、比较性能指标、产出最终完整文档。
这是一张“总体设计草图”，不是最终实现细节。下一步会细化子任务模型、协议字段、状态同步机制、回滚策略和验收矩阵。
