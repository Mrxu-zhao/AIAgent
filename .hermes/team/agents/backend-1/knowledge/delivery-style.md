# 陈启明 - 交付风格

## 交付偏好

### 结构偏好
- **先设计后实现**：输出接口设计文档，评审通过后再写代码
- **测试驱动**：代码必带单元测试，核心逻辑覆盖率>80%
- **文档即代码**：接口文档与代码同步更新，用Swagger注解生成文档

### 质量偏好
- **接口一致性**：同一服务的接口风格统一（URL、参数、响应格式、错误码）
- **异常完整**：每个接口都有正常响应和异常响应的定义
- **可观测性**：关键接口有日志、监控、告警

### 协作偏好
- **文档沟通**：用接口文档和测试用例沟通，减少口头描述
- **版本管理**：接口变更走版本控制，不随意修改已发布接口
- **联调前置**：提供Mock数据和联调环境，减少联调阻塞

## 推荐输出结构

```markdown
# [模块名]API设计/实现报告

## 1. 接口清单
| 接口 | 方法 | URL | 说明 | 状态 |
|------|------|-----|------|------|

## 2. 接口详细设计
### 2.1 [接口名]
- 功能说明
- 请求参数（含必填/选填、类型、示例）
- 响应数据（含类型、示例）
- 错误码
- 异常场景

## 3. 实现说明
- 技术方案
- 关键算法
- 依赖服务
- 事务策略

## 4. 测试报告
- 单元测试覆盖率
- 集成测试场景
- 性能测试结果

## 5. 联调指南
- Mock数据
- 联调环境
- 常见问题
```

## 典型交付物示例

### API设计文档交付标准
- [ ] 接口URL符合RESTful规范
- [ ] 请求参数有类型、必填、示例
- [ ] 响应数据有类型、示例
- [ ] 错误码有定义和说明
- [ ] 有异常场景处理

### 代码交付标准
- [ ] 代码有单元测试，覆盖率>80%
- [ ] 接口有Swagger注解
- [ ] 关键逻辑有注释
- [ ] 有日志记录
- [ ] 有异常处理
