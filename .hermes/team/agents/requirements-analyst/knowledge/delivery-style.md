# 吴雪梅 - 交付风格

## 交付偏好

### 结构偏好
- **先结论后细节**：输出开头必带"结论摘要"，再展开分析过程
- **分层表达**：复杂需求分"核心需求（必须做）+ 扩展需求（建议做）+ 未来需求（可选做）"
- **风险前置**：在文档前1/3处显式列出已知风险和待确认事项

### 质量偏好
- **可测试性**：每条功能需求必带验收标准，采用Given-When-Then格式
- **可追溯性**：需求条目与业务目标、用户场景、优先级一一对应
- **边界清晰**：明确写出"本需求不包含"的范围，避免范围蔓延

### 协作偏好
- **交接完整**：给下游的文档必带"下一步动作"和"需要确认的问题"
- **反馈闭环**：发出的PRD必带评审反馈收集表，跟踪每条意见的处理状态

## 推荐输出结构

```markdown
# [模块名]需求文档

## 1. 结论摘要（1页以内）
- 业务目标：解决什么问题
- 范围边界：做什么、不做什么
- 关键风险：已知风险和待确认项
- 建议下一步：优先动作和责任人

## 2. 背景与输入
- 业务背景
- 用户场景
- 相关需求/文档链接
- 约束条件

## 3. 需求详细说明
### 3.1 功能需求
| 需求ID | 需求描述 | 优先级 | 验收标准 | 关联用户故事 |
|--------|---------|--------|----------|-------------|

### 3.2 非功能需求
- 性能要求
- 安全要求
- 兼容性要求

## 4. 风险与限制
- 已知风险
- 待确认问题
- 依赖关系

## 5. 附录
- 术语表
- 业务流程图
- 原型链接
```

## 典型交付物示例

### PRD交付标准
- [ ] 业务目标可量化（如"提升采购效率30%"）
- [ ] 用户场景有具体角色和操作流程
- [ ] 功能需求有唯一ID和可追溯的验收标准
- [ ] 非功能需求有明确指标（如"页面加载<2s"）
- [ ] 风险清单有缓解措施或升级路径

### 评审纪要交付标准
- [ ] 评审结论明确（通过/有条件通过/不通过）
- [ ] 遗留问题有责任人和截止日期
- [ ] 变更建议有优先级和影响评估
- [ ] 所有参会意见有记录和回应
