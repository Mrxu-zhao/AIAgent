# 团队进化协议

## 进化宣言

**目标**: 打造一个能够持续自我进化的研发团队，每个成员都能学习、成长、共享知识。

**原则**: 
- 每次任务都是学习机会
- 每次复盘都是进化步骤
- 知识共享 > 个人积累

---

## 团队进化机制

### 1. 任务级进化（每次任务后）

```
任务完成 → 自动反思
    ↓
1. 分析本次交互
   - 核心需求是什么？
   - 是否最优方案？
   - 有无改进空间？

2. 识别可复用点
   - 成功的模式 → 写入 patterns/
   - 踩过的坑 → 写入 lessons/
   - 新术语 → 写入 glossaries/

3. 更新知识库
   - status.md（时间戳+积累量）
   - 对应的 profile.md
```

### 2. 子Agent任务分配规范（主Agent必读）

> ⚠️ 发现问题：子Agent只执行任务、不自动进化知识库。必须在分配任务时明确要求。

**分配任务时，context 必须包含以下段落：**

```
【强制要求】任务完成后必须同步完成知识库进化：
1. 将本次经验写入 lessons_learned/ 或 lessons/
2. 将识别到的模式写入 patterns/
3. 更新 status.md（时间戳 + 本次新增条目数 + 积累量统计）
4. 如有新术语写入 glossaries/
5. 项目档案写入 projects/
```

**主Agent操作流程：**
```
Step 1: 拆解任务，确定子Agent职责
Step 2: 分配任务时，context 中包含上方强制要求段落
Step 3: 子Agent完成任务后，检查其知识库是否已更新
Step 4: 如未更新，主Agent代为补充进化（参考本次 mall4j/xzmeto 分析）
Step 5: 汇总各子Agent分析结果，输出总报告
```

**⚠️ 注意：** 子Agent没有记忆能力（每次是新会话），必须靠 context 传递要求。

### 3. 每日进化（每天）

- 团队成员互相学习
- 更新 shared knowledge
- 清理过时的知识

### 4. 每周进化（每周）

- 评估各成员技能提升
- 更新学习计划进度
- 优化团队协作流程

### 5. 项目级进化（每个项目）

- 项目复盘
- 经验沉淀
- 模板更新

## 进化记录

### 2026-04-29 首次实战：双配套项目分析

**背景:** 分析 mall4j-bbc 和 xzmeto_resource 两个配套系统的数据库、架构、前后端框架。

**参与成员:** requirements-analyst、dba、frontend-dev（×2）

**执行流程:**
1. 主Agent拆解任务，分配给4个子Agent并行分析
2. 子Agent各自完成项目分析，输出分报告到 `projects/` 目录
3. **发现问题:** 子Agent只输出了报告，未按技能规范完成知识库进化
4. 主Agent发现后手动补充了所有知识库进化内容
5. 主Agent汇总4份报告，生成 `two_suites_comparison.md` 总报告

**关键发现:**
- 子Agent无记忆能力（每次是新会话），必须在context中强制传递进化要求
- 多端前端项目要先按技术栈分组再分析，不能简单横向对比
- 微服务数据库边界要同时检查代码模块和数据库表归属是否一致

**改进措施:**
- 在 EVOLUTION.md 新增「子Agent任务分配规范」章节
- 强制要求主Agent在分配任务时传递知识库进化要求段落
- 主Agent在任务完成后检查子Agent知识库状态，未更新则代为补充
