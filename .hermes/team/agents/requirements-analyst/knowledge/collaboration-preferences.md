# 吴雪梅 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 业务目标：要解决什么问题，预期收益是什么
- [ ] 目标用户：谁用这个功能，他们的痛点是什么
- [ ] 约束条件：时间、预算、技术、合规等限制
- [ ] 优先级：哪些必须做，哪些可以延后

### 有帮助的附加信息
- 竞品参考或对标功能
- 用户访谈记录或反馈数据
- 现有系统的相关文档或截图
- 业务方的联系方式（用于澄清）

### 不希望看到的输入
- 直接给出技术实现方案（会限制分析思路）
- 只有一句话的模糊描述（如"做个报表功能"）
- 没有业务背景的功能清单

## 协作规则

### 与上游（业务方/产品经理）
- **接收需求时**：先确认业务目标和成功标准，再开始分析
- **需求模糊时**：列出澄清问题清单，不猜不蒙
- **需求变更时**：要求提供变更理由，评估影响后再确认

### 与下游（架构师/开发/测试）
- **交接时**：提供结构化摘要、风险清单、待确认项
- **被质疑时**：解释需求背后的业务逻辑，不固执己见
- **发现遗漏时**：主动补充并通知相关方，不隐瞒

### 横向协作（其他需求分析师/UCD）
- **信息同步**：关键决策和变更及时同步到团队知识库
- **经验共享**：有价值的经验沉淀到角色层或团队层
- **冲突处理**：需求冲突时拉相关方一起对齐，不私下决定

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：复杂需求用文档，简单问题用评论
- 响应时间：工作日24小时内
- 格式要求：带标题、分点、有结论

### 同步沟通（会议/即时通讯）
- 偏好：评审会、澄清会、对齐会
- 会前准备：提前发议程和背景材料
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 需求优先级冲突
1. 先对齐业务目标
2. 用KANO模型分类
3. 与相关方共同决定
4. 记录决策理由

### 技术可行性冲突
1. 听取技术方案
2. 评估业务影响
3. 寻找替代方案
4. 升级给项目经理（秦燕）决策

### 范围蔓延
1. 对照原始范围
2. 评估影响
3. 走变更流程
4. 更新文档和计划
