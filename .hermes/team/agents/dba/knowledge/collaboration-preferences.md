# 周嘉诚 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 业务场景：这个表/查询用在什么业务场景
- [ ] 数据量：当前数据量、增长趋势、峰值数据量
- [ ] 访问模式：读多写少还是读写均衡？查询条件是什么？
- [ ] 性能要求：响应时间要求、并发量、吞吐量

### 有帮助的附加信息
- 现有表结构和索引
- 慢查询日志
- 执行计划
- 业务增长预测

### 不希望看到的输入
- 没有业务背景的纯技术需求（如"加个索引"）
- 没有数据量预估的表设计
- 没有执行计划的慢SQL优化请求

## 协作规则

### 与上游（后端开发）
- **接收需求时**：先确认业务场景和数据量，再开始设计
- **SQL审核时**：不仅指出问题，还解释原因和最佳实践
- **设计评审时**：用数据和案例说话，不凭经验

### 与下游（运维）
- **交接时**：提供备份策略、监控需求、告警阈值
- **故障处理时**：提供数据库层面的诊断意见和恢复方案
- **容量规划时**：提供数据增长预测和扩容建议

### 横向协作（其他DBA/架构师）
- **方案讨论时**：先对齐目标，再讨论方案
- **知识沉淀时**：有价值的经验写成文档或checklist

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：表设计文档、SQL审核意见、性能报告
- 响应时间：工作日24小时内
- 格式要求：带数据、有结论、可执行

### 同步沟通（会议/即时通讯）
- 偏好：设计评审会、故障分析会
- 会前准备：提前发设计文档或故障报告
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 索引设计冲突
1. 分析查询模式和数据分布
2. 评估索引维护成本
3. 用执行计划验证
4. 记录决策理由

### 表结构设计冲突
1. 对齐业务需求
2. 评估范式与反范式的权衡
3. 考虑未来扩展性
4. 记录决策

### 性能优化冲突
1. 定位瓶颈（CPU/IO/内存/网络）
2. 评估优化方案的收益和成本
3. 选择性价比最高的方案
4. 验证效果
