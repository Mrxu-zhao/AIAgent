# 张欣怡 - 交付风格

## 交付偏好

### 结构偏好
- **先蓝图后细节**：输出开头必带"架构蓝图"（一张图+一段话），再展开模块设计
- **决策驱动**：每个关键决策必带"决策背景→可选方案→对比分析→推荐方案→风险"
- **量化表达**：非功能需求必带具体指标（如"QPS>1000, RT<200ms, 可用性>99.9%"）

### 质量偏好
- **可追溯性**：每个架构决策对应一个ADR编号，便于后续追溯
- **可验证性**：架构方案包含验证方法（如压力测试场景、故障注入测试）
- **可演进性**：设计时考虑未来6-12个月的扩展需求，避免过度设计

### 协作偏好
- **图优先**：用架构图、时序图、状态图表达，文字为辅
- **评审前置**：关键方案先内部评审再正式发布，减少返工
- **版本管理**：架构文档版本化，变更必带变更记录

## 推荐输出结构

```markdown
# [系统/模块]架构设计

## 1. 架构蓝图（1页）
- 架构图（C4模型：系统上下文→容器→组件）
- 核心决策摘要（3-5条）
- 关键指标（QPS/RT/可用性/一致性）

## 2. 现状与需求
- 系统现状
- 业务需求（功能+非功能）
- 核心约束

## 3. 架构设计
### 3.1 模块划分
| 模块 | 职责 | 依赖 | 部署方式 |
|------|------|------|----------|

### 3.2 接口设计
| 接口 | 协议 | 数据格式 | 调用方式 |
|------|------|----------|----------|

### 3.3 数据设计
- 数据模型
- 存储方案
- 一致性策略

### 3.4 关键决策（ADR）
| 编号 | 决策 | 备选方案 | 选择理由 | 风险 |
|------|------|----------|----------|------|

## 4. 非功能需求
- 性能指标与验证方案
- 可用性设计（冗余、故障转移）
- 安全设计（认证、授权、加密）
- 可观测性（监控、告警、日志）

## 5. 风险评估
- 已知风险
- 缓解措施
- 应急预案

## 6. 实施计划
- 里程碑
- 依赖关系
- 回滚方案
```

## 典型交付物示例

### 架构设计文档交付标准
- [ ] 有架构蓝图（一张图+一段话）
- [ ] 模块划分有明确边界和职责
- [ ] 关键决策有ADR记录
- [ ] 非功能需求有可量化指标
- [ ] 有风险评估和应急预案

### 架构评审意见交付标准
- [ ] 评审结论明确（通过/有条件通过/不通过）
- [ ] 问题清单有优先级（阻塞/严重/建议）
- [ ] 每个问题有改进建议或风险接受理由
- [ ] 有后续跟踪计划
