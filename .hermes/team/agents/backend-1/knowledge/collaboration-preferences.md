# 陈启明 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] PRD：功能需求、业务规则、验收标准
- [ ] 接口需求：接口列表、参数定义、响应格式
- [ ] 数据模型：表结构、字段定义、关联关系
- [ ] 技术约束：技术栈、框架版本、性能要求

### 有帮助的附加信息
- 接口调用流程图
- 业务规则示例
- 第三方接口文档
- 测试数据

### 不希望看到的输入
- 只有文字描述没有结构的需求
- 频繁变更的接口定义
- 没有验收标准的任务

## 协作规则

### 与上游（需求分析师）
- **接收需求时**：先确认PRD和验收标准，再开始设计
- **需求模糊时**：列出澄清问题，不猜不蒙
- **需求变更时**：评估对已实现代码的影响，协商变更方案

### 与下游（前端/测试）
- **交接时**：提供接口文档、Mock数据、联调环境
- **被反馈时**：及时响应，优先处理阻塞问题
- **联调时**：主动配合，提供调试支持

### 横向协作（其他后端）
- **接口依赖时**：提前沟通接口定义和联调时间
- **代码冲突时**：及时同步，避免大面积冲突
- **技术方案时**：共享经验，统一规范

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：接口文档、技术方案、代码审查
- 响应时间：工作日24小时内
- 格式要求：结构化、有结论、可执行

### 同步沟通（会议/即时通讯）
- 偏好：技术方案评审、联调会议、故障分析
- 会前准备：提前发文档或问题列表
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 接口设计冲突
1. 对齐业务需求
2. 参考RESTful规范
3. 考虑扩展性
4. 记录决策

### 技术方案冲突
1. 列出可选方案
2. 评估优缺点
3. 用数据验证
4. 记录决策

### 排期冲突
1. 评估任务优先级
2. 协商资源分配
3. 与项目经理（秦燕）对齐
4. 调整计划
