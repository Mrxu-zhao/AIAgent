# Repository Agent Discipline 设计文档

## 1. 背景

当前仓库已经有面向 Hermes 团队内部协作的知识与角色约束，例如：

- 根目录 `README.md` 说明仓库主线和当前推荐入口。
- `CODE_WIKI.md` 对仓库分层、主目录和关键实现做了整体说明。
- `.hermes/SOUL.md`、各角色 `SOUL.md`、`config.yaml`、`SKILL.md` 约束 Hermes 体系内 agent 的行为。
- `.hermes/team/knowledge/` 提供团队公共知识入口和协作流程说明。

但这些约束主要服务于仓库内部的 Hermes/团队 agent，不足以约束“外部进入仓库执行修改任务的大模型”。这类模型在实际协作中容易出现几个典型问题：

- 没有先读完整相关代码，就直接给方案或根因判断。
- 只看局部目录或历史实现，就把它当成当前主线。
- 用经验和训练语料脑补项目结构，而不是引用真实文件事实。
- 给出大而空的建议，缺少“已读证据、方案落点、验证方式、剩余未知项”。

因此需要在仓库根目录新增一份对外显式可见的 `AGENTS.md`，作为进入仓库工作的统一规约文件，让其他大模型在分析、设计、修改和评审这个 AI Agent 项目时遵循一致的最低工作标准。

## 2. 目标与非目标

### 2.1 目标

- 在仓库根目录提供一份对外可见、模型友好的 `AGENTS.md`。
- 明确要求模型先读仓库事实，再输出方案或修改建议。
- 强制模型在输出中包含“已读文件、确认事实、修改计划、验证方式、未确认项”。
- 将本仓库的真实主线、关键入口和旧实现边界写清楚，减少模型走偏。
- 用一条最小回归测试保证 `AGENTS.md` 持续存在且包含关键锚点。

### 2.2 非目标

- 本轮不修改 `.hermes` 内部 Hermes 角色配置，不重写角色 `SOUL.md` 或 `SKILL.md`。
- 本轮不引入运行时强制门禁，不修改 control plane 的执行逻辑。
- 本轮不新增 `agent.md`、多份镜像文档或外部平台专用提示词。

## 3. 方案比较

### 方案 A：只补一份通用提示词文档

在根目录新增一份简短文档，只写“先看代码、不要脑补、改完要测试”等原则。

优点：

- 改动最小。
- 几乎没有维护成本。

缺点：

- 过于抽象，模型容易表面遵守。
- 没有绑定本仓库的真实结构和主线入口。
- 无法有效约束输出格式。

### 方案 B：新增仓库级 `AGENTS.md`，包含强制流程和输出契约

在根目录新增 `AGENTS.md`，同时写明：

- 进入仓库后必须先读哪些文件。
- 哪些目录是主线，哪些目录是兼容层。
- 不同任务类型输出时必须包含哪些证据和结论结构。
- 哪些行为禁止发生。

优点：

- 与多数代码模型/agent 工具兼容。
- 既有行为约束，也有输出契约。
- 能把这个仓库的事实锚点直接暴露给外部模型。

缺点：

- 需要控制篇幅，否则模型可能跳读。

### 方案 C：同时新增 `AGENTS.md`、`agent.md` 和多份子文档

为兼容不同工具，新增多份入口文件，再拆出多个补充说明文档。

优点：

- 覆盖面更广。

缺点：

- 容易出现内容漂移。
- 第一版维护成本偏高，且没有明确证据表明本仓库当前需要双文件策略。

### 方案选择

本轮选择 **方案 B：新增仓库级 `AGENTS.md`，包含强制流程和输出契约**。

## 4. 设计原则

- **短而硬**：核心规则必须集中、直接、可执行，避免写成长篇散文。
- **先证据后判断**：没有已读文件和确认事实，就不应给最终结论。
- **仓库事实优先**：一切项目判断以真实文件为准，而不是训练经验。
- **主线明确**：必须显式告诉模型当前主线和兼容层边界。
- **输出可检查**：让模型按固定结构回应，降低“看起来很努力、实际上没读代码”的空间。

## 5. `AGENTS.md` 内容设计

### 5.1 文件定位

`AGENTS.md` 放在仓库根目录，面向所有进入本仓库工作的模型和 agent。

### 5.2 结构

第一版采用以下结构：

1. 文件使命
2. 必须先读的仓库入口
3. 强制工作流程
4. 输出契约
5. 仓库事实锚点
6. 禁止事项
7. 修改与验证要求
8. AI Agent 项目专项检查清单

### 5.3 强制流程

`AGENTS.md` 中要求模型默认按以下顺序工作：

1. 先读 `README.md` 和 `CODE_WIKI.md`。
2. 再确认任务相关目录、主实现和兼容层边界。
3. 再读实际相关文件，而不是只看目录名。
4. 再输出方案、评审意见或修改计划。
5. 修改后运行相关验证，并报告结果。

### 5.4 输出契约

`AGENTS.md` 中要求模型在分析/方案/修改类任务中至少输出：

- 已读文件
- 从代码确认的事实
- 准备改动的文件
- 验证方式
- 未确认项或风险

这样可以把“先探索、再结论”的行为显式化。

### 5.5 仓库事实锚点

`AGENTS.md` 中需要明确如下项目事实：

- 当前推荐主入口是 `.hermes/team/control_plane/`。
- `.hermes/team/调度框架/` 仍存在，但属于历史调度框架与兼容入口。
- 涉及 agent 行为、知识装载或角色约束时，必须同时检查：
  - `.hermes/SOUL.md`
  - `.hermes/agents/*/SOUL.md`
  - `.hermes/agents/*/config.yaml`
  - `.hermes/skills/*/SKILL.md`
  - `.hermes/team/knowledge/`

## 6. 测试策略

在 `tests/control_plane/test_documentation_contracts.py` 中新增一条契约测试，验证根目录 `AGENTS.md`：

- 文件存在。
- 包含 `README.md`、`CODE_WIKI.md`、`.hermes/team/control_plane/`、`.hermes/team/调度框架/` 等关键仓库锚点。
- 包含“已读文件”“确认事实”“验证”这类输出契约关键词。

## 7. 风险与缓解

- 风险：文档过长，模型跳读。
  - 缓解：控制结构层级，优先写硬规则与模板，不堆背景说明。
- 风险：仓库未来主线变化，`AGENTS.md` 失真。
  - 缓解：用文档契约测试锁定关键锚点，后续调整时同步更新测试。
- 风险：规则过泛，仍不足以抑制偷懒。
  - 缓解：通过固定输出结构，把“已读证据”变成每次回答都要暴露的字段。

## 8. 结论

本轮以根目录 `AGENTS.md` 作为“外部大模型进入仓库的第一份工作规约”，并配合最小文档契约测试，使其既能直接发挥约束作用，也能在后续演进中保持稳定存在和关键内容不漂移。
