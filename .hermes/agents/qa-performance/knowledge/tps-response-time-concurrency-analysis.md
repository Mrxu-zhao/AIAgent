# TPS、响应时间、并发数、吞吐量的关系和分析

> 作者：孙美玲（性能测试工程师）  
> 创建时间：2026-04-29

## 1. 核心概念定义

### 1.1 四大核心指标

```
性能指标三角关系:

    并发数 (C)
        ↓↑
    ┌─────────┐
    │   TPS   │ ← 吞吐量 (Throughput)
    └─────────┘
        ↓↑
    响应时间 (RT)
```

| 指标 | 定义 | 单位 | 说明 |
|------|------|------|------|
| **TPS** | Transactions Per Second，每秒事务数 | 个/秒 | 系统处理能力的核心指标 |
| **RT** | Response Time，响应时间 | ms | 用户感知的时间 |
| **C** | Concurrency，并发数 | 个 | 同时在系统的用户数 |
| **Throughput** | 吞吐量 | 请求/秒 | 资源处理能力 |

### 1.2 指标详解

**响应时间 (RT) 组成：**
```
响应时间 = 网络时间 + 服务器处理时间 + 数据库时间 + 渲染时间

示例:
  网络延迟:     10ms
  Web服务器:    20ms
  应用服务:     50ms  
  数据库:       30ms
  其他:         10ms
  ─────────────────
  总响应时间:   120ms
```

**TPS 与 QPS 的区别：**
| 指标 | 全称 | 说明 |
|------|------|------|
| TPS | Transactions Per Second | 业务事务（可能包含多个HTTP请求） |
| QPS | Queries Per Second | 查询数/秒（通常指单个HTTP请求） |
| RPS | Requests Per Second | 请求数/秒（同QPS） |
| HPS | Hits Per Second | 点击数/秒 |

```
关系: TPS = QPS (当1个事务=1个请求时)
      TPS < QPS (当1个事务>1个请求时)
      
示例: 
  用户下单事务 = 登录(1) + 查询商品(2) + 创建订单(1) = 4个请求
  如果QPS=400，则TPS=100
```

## 2. 核心公式

### 2.1 基础公式

```
公式1: TPS = 并发数 / 平均响应时间
       C = TPS × RT

公式2: Throughput = 并发数 × (1000 / 平均响应时间)
       = C × 1000 / RT

公式3: 并发用户数 = TPS × 平均响应时间 × 并发因子
       (并发因子通常 2-5，取决于业务特性)
```

### 2.2 公式推导

```
已知:
  - 100个并发用户
  - 平均响应时间: 200ms
  
求: TPS = ?

计算:
  每个用户每秒可处理: 1000ms / 200ms = 5个请求
  100个用户每秒共处理: 100 × 5 = 500个请求
  TPS = 500
```

### 2.3 实际应用计算

**场景1: 预估系统容量**

```
需求:
  - 目标: 支持10000用户同时在线
  - 用户操作比例: 每5秒操作1次
  - 每次操作产生3个请求
  
计算:
  操作频率 = 1 / 5 = 0.2 次/秒/用户
  在线用户并发 = 10000 × 0.2 = 2000 并发操作
  请求QPS = 2000 × 3 = 6000 QPS
  假设响应时间200ms:
  需要线程数 = 6000 × 0.2 = 1200 线程
```

**场景2: 确定需要的并发数**

```
需求:
  - 系统需要处理 1000 TPS
  - 平均响应时间: 500ms
  
计算:
  需要的并发数 = TPS × RT = 1000 × 0.5 = 500 并发用户
  
  考虑用户思考时间(3秒):
  用户活动因子 = 0.5 / 3.5 = 0.143
  在线用户数 = 1000 / 0.143 ≈ 7000 用户
```

## 3. 性能曲线分析

### 3.1 理想状态

```
TPS与并发数关系 (理想状态):

TPS
 ↑
 │           ╱  (线性增长)
 │         ╱
 │       ╱
 │     ╱
 │   ╱
 │ ╱____________
 └─────────────────→ 并发数
    0        拐点
```

### 3.2 实际状态

```
实际性能曲线:

TPS
 ↑
 │      ╱‾‾‾‾‾╲        ← 拐点出现
 │    ╱        ╲
 │  ╱           ╲
 │╱              ╲____   ← 下降
 └──────────────────────→ 并发数
   拐点  最大容量  崩溃区
```

### 3.3 拐点分析

```
拐点前 (线性区):
  - 资源充足
  - TPS与并发数成正比
  - 响应时间稳定
  - 无明显瓶颈

拐点处 (过渡区):
  - 资源开始紧张
  - TPS增长放缓
  - 响应时间开始上升
  - 需要监控定位瓶颈

拐点后 (饱和区):
  - 资源耗尽
  - TPS不再增长或下降
  - 响应时间急剧上升
  - 系统接近崩溃
```

### 3.4 响应时间曲线

```
响应时间随并发变化:

RT(ms)
 ↑
 │                ← 急剧上升
 │              ╱
 │            ╱
 │          ╱
 │        ╱
 │      ╱
 │    ╱___________  ← 轻微上升
 │  ╱
 │╱_____________→ 并发数
```

## 4. 性能分析模型

### 4.1 性能分析金字塔

```
            ┌─────────┐
            │  业务层  │  ← 用户体验 (响应时间)
            ├─────────┤
            │ 应用层   │  ← JVM、连接池、线程池
            ├─────────┤
            │   DB层   │  ← SQL优化、索引、连接池
            ├─────────┤
            │   OS层   │  ← CPU、内存、磁盘、网络
            └─────────┘
```

### 4.2 分析维度

| 维度 | 关注指标 | 分析方法 |
|------|----------|----------|
| 用户视角 | RT、错误率 | APM、用户体验监控 |
| 应用视角 | TPS、线程数 | JMeter、监控平台 |
| 资源视角 | CPU、内存、IO | 系统监控 |
| 数据库视角 | SQL耗时、连接数 | DB监控、慢查询日志 |

### 4.3 性能瓶颈定位

```
定位流程:

1. 发现问题
   响应时间 > 目标 或 TPS < 目标
   
2. 确认范围
   - 单接口慢 还是 全局慢
   - 规律性慢 还是 偶发性慢
   - Load高 还是 响应时间长
   
3. 分层排查
   ┌─────────────────────────────────┐
   │ ① 前端: F12 Network、APM        │
   ├─────────────────────────────────┤
   │ ② 网关: 请求耗时分布              │
   ├─────────────────────────────────┤
   │ ③ 应用: 线程Dump、CPU Profile   │
   ├─────────────────────────────────┤
   │ ④ DB: 慢查询、连接池             │
   ├─────────────────────────────────┤
   │ ⑤ 系统: top、vmstat、iostat     │
   └─────────────────────────────────┘
   
4. 定位根因
   - CPU高 → 代码问题? GC?
   - 内存高 → 内存泄漏? 缓存过大?
   - IO高 → 磁盘? 网络带宽?
   - 连接池满 → 慢SQL? 连接泄漏?
```

## 5. 性能指标计算示例

### 5.1 电商系统容量评估

```yaml
业务背景:
  目标用户: 100万
  高峰时段活跃用户: 20%
  高峰TPS需求: 5000
  
业务比例:
  - 浏览商品: 60%
  - 搜索查询: 20%
  - 下单支付: 15%
  - 其他操作: 5%
  
单业务响应时间(参考):
  - 浏览商品: 150ms
  - 搜索查询: 300ms
  - 下单支付: 800ms
  
计算:
  高峰在线用户 = 1000000 × 20% = 200000
  用户思考时间 = 5秒
  操作频率 = 1/5 = 0.2次/秒/用户
  
  TPS = 200000 × 0.2 = 40000 QPS
  
  各业务TPS:
  - 浏览: 40000 × 60% = 24000 TPS
  - 搜索: 40000 × 20% = 8000 TPS
  - 下单: 40000 × 15% = 6000 TPS
  
  需要的并发数:
  - 浏览: 24000 × 0.15 = 3600
  - 搜索: 8000 × 0.3 = 2400
  - 下单: 6000 × 0.8 = 4800
  - 总并发 = 3600 + 2400 + 4800 = 10800
```

### 5.2 JMeter 线程数计算

```yaml
目标: 单接口压测达到1000 TPS

已知:
  - 单请求平均响应时间: 200ms
  
计算:
  需要的并发数 = TPS × RT = 1000 × 0.2 = 200 线程
  
  JMeter配置:
  Thread Group:
    Number of Threads: 200
    Ramp-Up Period: 30      (30秒内启动完成)
    Loop Count: Forever
    Duration: 300           (持续5分钟)
    
  验证:
  实际TPS ≈ 200 / 0.2 = 1000 ✓
```

## 6. 性能指标监控

### 6.1 监控指标体系

```
关键性能指标 (KPI):

业务层:
├── 每秒请求数 (QPS)
├── 每秒事务数 (TPS)
├── 平均响应时间 (ART)
├── 95/99分位响应时间 (P95/P99)
├── 错误率 (Error Rate)
└── 超时率 (Timeout Rate)

应用层:
├── JVM Heap 使用率
├── Full GC 频率
├── 活跃线程数
├── 连接池使用率
└── 缓存命中率

数据库层:
├── QPS
├── 慢查询数量
├── 连接池使用率
├── 主从延迟
└── 锁等待时间

系统层:
├── CPU 使用率
├── 内存使用率
├── 磁盘IO
├── 网络IO
└── 负载 (Load Average)
```

### 6.2 性能基线

```yaml
性能基线标准 (参考值):

等级        | RT(平均) | RT(P99)  | TPS   | 错误率
------------|----------|----------|-------|--------
Excellent   | < 200ms  | < 500ms  | > 1000| < 0.1%
Good        | < 500ms  | < 1s     | > 500 | < 0.5%
Acceptable  | < 1s     | < 2s     | > 100 | < 1%
Warning     | < 2s     | < 5s     | > 50  | < 5%
Critical    | > 2s     | > 5s     | < 50  | > 5%
```

### 6.3 性能测试报告指标

```yaml
测试报告必含指标:

1. 基础信息
   - 测试时间
   - 测试环境配置
   - 测试数据量
   - JMeter配置

2. 核心指标
   - TPS 趋势图
   - 响应时间分布图
   - 并发用户数
   - 成功率

3. 详细数据
   - 各接口RT统计
   - TPS统计
   - 错误统计
   - 资源使用情况

4. 结论
   - 是否满足性能要求
   - 瓶颈分析
   - 优化建议
```
