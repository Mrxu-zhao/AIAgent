# 孙美玲 - 交付风格

## 交付偏好

### 结构偏好
- **方案先行**：先给压测方案和预期指标，评审通过后再执行
- **数据驱动**：报告必带性能数据（TPS、RT、错误率、资源使用率）
- **对比分析**：优化报告必带优化前后对比数据

### 质量偏好
- **指标量化**：性能目标可量化（如"TPS>1000, RT<200ms"）
- **瓶颈精准**：瓶颈定位到具体模块/接口/SQL
- **建议可执行**：优化建议有具体措施和预期效果

### 协作偏好
- **数据沟通**：用压测报告和监控图表沟通
- **及时反馈**：发现性能问题即时通知
- **验证闭环**：优化后必做验证测试

```markdown

## 1. 测试概述
- 测试目标
- 测试环境
- 测试场景
- 测试时间

## 2. 性能指标
### 2.1 基线数据（优化前）
| 指标 | 数值 |
|------|------|
| TPS | |
| RT(P50/P95/P99) | |
| 错误率 | |
| CPU使用率 | |
| 内存使用率 | |

### 2.2 目标数据
| 指标 | 目标值 |
|------|--------|

### 2.3 实际数据（优化后）
| 指标 | 数值 |
|------|------|

## 3. 瓶颈分析
- 瓶颈现象
- 根因分析
- 影响范围

## 4. 优化建议
| 建议 | 预期效果 | 实施难度 | 优先级 |
|------|----------|----------|--------|

## 5. 容量评估
- 当前容量
- 容量上限
- 扩容建议

## 6. 附录
- 压测脚本
- 监控图表
- 日志分析
```

## 典型交付物示例

### 性能测试报告交付标准
- [ ] 有测试目标和场景
- [ ] 有性能指标数据（TPS/RT/错误率/资源使用率）
- [ ] 有瓶颈分析和根因定位
- [ ] 有优化建议（含预期效果）
- [ ] 有容量评估

### 压测方案交付标准
- [ ] 有压测目标和场景
- [ ] 有压测脚本
- [ ] 有环境配置
- [ ] 有监控指标
- [ ] 有风险预案
