# 测试漏测教训总结

> 作者：郑晓彤 | 创建时间：2026-04-29 | 用途：记录测试漏测教训，避免重蹈覆辙

## 1. 案例一：分页边界值漏测

### 问题描述
上线后发现，当列表数据刚好等于分页大小时，查询结果异常。

### 教训
之前只测试了"有数据"和"无数据"两种情况，忽略了刚好等于分页大小的边界场景。

### 正确做法
```
边界值测试必须包含：
- 0条数据
- 1条数据（小于pageSize）
- 刚好等于pageSize
- pageSize+1条数据
- 2*pageSize条数据
- 最大页码
- 最大页码+1
```

### 代码级检查点
```java
// MyBatis-Plus分页必须验证
Page<User> page = new Page<>(pageNum, pageSize);
List<User> list = userMapper.selectPage(page, null);

// 测试必须验证
assertEquals(pageSize, list.size()); // 当total > pageSize时
assertEquals(total % pageSize, list.size()); // 最后一页
```

## 2. 案例二：权限验证漏测

### 问题描述
用户可以通过修改URL直接访问其他用户的订单详情，发生了越权漏洞。

### 教训
- 只测试了"未登录不能访问"
- 没有测试"登录用户访问别人的数据会怎样"

### 正确做法
```
权限测试必须包含：
□ 未登录访问 → 401
□ 普通用户访问管理员URL → 403
□ 用户A访问用户B的数据 → 403或无数据
□ 通过URL直接访问其他用户ID → 正确拒绝
□ 通过API直接调用 → 接口层面也要校验
```

### 教训总结
```
页面级别测试 ≠ 接口级别测试
- 页面隐藏按钮不代表接口安全
- 必须直接调用接口测试权限
```

## 3. 案例三：并发场景漏测

### 问题描述
两个管理员同时审批同一个订单，都显示审批成功，但最终状态异常。

### 教训
只测试了单人单次操作，没有测试多人同时操作同一数据的场景。

### 正确做法
```
并发测试场景：
1. 库存超卖测试
   - 库存=1时，10人同时下单
   - 结果：只能1人成功
   
2. 重复提交测试
   - 快速点击提交按钮2次
   - 结果：只创建1条记录
   
3. 并发审批测试
   - 2人同时审批同一申请
   - 结果：只有1人成功
   
4. 并发修改测试
   - 2人同时编辑同一记录
   - 结果：后提交的覆盖先提交的（需确认是否期望）
```

## 4. 案例四：日期边界漏测

### 问题描述
12月31日创建的数据，按月度统计时没有被统计进去。

### 教训
只测试了普通日期，没有测试跨年、跨月、闰年等边界日期。

### 正确做法
```
日期边界测试场景：
□ 普通工作日
□ 周末
□ 月初第一天
□ 月末最后一天
□ 跨月：12月31日 → 1月1日
□ 闰年：2月29日在闰年/非闰年
□ 年度边界：12月31日 vs 1月1日
□ 时区差异（UTC vs 本地）
□ 夏令时切换
□ 法定节假日（如果有特殊逻辑）
```

## 5. 案例五：前端校验 vs 后端校验混淆

### 问题描述
测试时只在前端输入非法值，被后端拦截没问题。但上线后有用户绕过前端直接调用API，发现后端没有校验。

### 教训
测试必须绕过前端直接测试接口，不能只依赖前端校验。

### 正确做法
```
接口测试要求：
1. 每个接口都单独测试
2. 绕过前端直接调用接口
3. 验证后端参数校验是否完整
4. 验证前端校验和后端校验是否一致
5. 使用Postman/curl测试接口
```

### 教训总结
```
前端校验 = 用户体验优化
后端校验 = 数据安全保证
两者缺一不可！
```

## 6. 案例六：分页参数越界漏测

### 问题描述
用户传入 pageSize=999999 导致数据库查询超时，系统卡死。

### 教训
只测试了正常分页参数，没有测试超大分页参数。

### 正确做法
```
分页参数必须验证：
- pageNum = 0, -1, 负数
- pageSize = 0, -1, 负数
- pageSize 超大值（999999）
- pageNum 超大值（999999）
- 非数字字符
- SQL注入尝试

后端必须做参数限制：
if (pageSize > 100) pageSize = 100;
```

## 7. 案例七：删除关联数据漏测

### 问题描述
删除用户后，与该用户关联的订单仍然显示"用户：张三"，数据不一致。

### 教训
只测试了用户删除成功，没有测试关联数据是否正确处理。

### 正确做法
```
删除操作必须验证：
□ 删除成功后数据真的不存在了
□ 关联查询不再返回该数据
□ 关联表的外键约束处理（级联删除/置空）
□ 删除后的数据恢复能力
□ 操作日志是否记录
```

## 8. 案例八：返回数据格式不一致

### 问题描述
列表有数据时返回 `{"data": []}`，列表无数据时返回 `{"data": null}`，前端处理不一致导致报错。

### 教训
没有统一测试空数据和非空数据的响应格式。

### 正确做法
```
响应格式一致性测试：
□ 有数据时：data 是数组 []
□ 无数据时：data 是空数组 []（不是 null）
□ 异常时：data 是 null，message 有错误信息
□ 分页数据：data 包含 list, total, pageNum 等字段
□ 确保 JSON.parse() 不会报错
```

## 9. 教训总结清单

### 测试前必须确认

```markdown
□ 功能需求文档是否完整？
□ 边界条件是否明确？
□ 异常处理是否定义？
□ 权限要求是否清晰？
□ 数据流转是否完整？
```

### 测试必须覆盖的场景

```markdown
□ 正常流程
□ 异常流程
□ 边界值（0、最大值、最大值+1）
□ 空数据
□ 单条数据
□ 大量数据
□ 并发操作
□ 权限校验
□ 参数格式校验
□ 安全校验（注入、XSS）
```

### 容易被忽略的测试点

```markdown
□ 直接URL访问绕过前端权限
□ 并发操作同一数据
□ 跨年/跨月/闰年日期边界
□ 超大参数值
□ 网络异常处理
□ Session/Cookie过期
□ 接口响应格式一致性
□ 删除操作的关联影响
□ 缓存与数据库一致性
□ 多环境差异（浏览器/OS）
```
