# 功能测试用例设计方法

> 作者：郑晓彤 | 创建时间：2026-04-29 | 适用：Java Spring Boot + Vue 3 B端管理后台

## 1. 等价类划分法

### 核心思想
将输入域划分为若干等价类，从每个类中选取少量代表性数据进行测试。

### 分类
| 类型 | 说明 | 示例 |
|------|------|------|
| 有效等价类 | 符合需求规范的输入 | 用户名 3-20 位字母数字 |
| 无效等价类 | 违反需求规范的输入 | 用户名包含特殊字符、长度超限 |

### 实战示例：登录功能

```
需求：用户名 3-20 位，密码 6-20 位

有效等价类：
  - 用户名：3位(aaa)、20位(20个字符)、中间长度(10位)
  - 密码：6位(123456)、20位(20个数字)、中间长度(12位)

无效等价类：
  - 用户名：2位(aa - 少于最小值)
  - 用户名：21位(21个字符 - 超过最大值)
  - 用户名：包含空格、特殊字符(!@#$%)
  - 密码：5位(少于最小值)
  - 密码：21位(超过最大值)
  - 用户名为空
  - 密码为空
```

### 代码参考
```java
// 后端校验规则（供测试参考）
@Size(min = 3, max = 20, message = "用户名长度3-20位")
@Pattern(regexp = "^[a-zA-Z0-9]+$", message = "用户名只能包含字母数字")
private String username;

@Size(min = 6, max = 20, message = "密码长度6-20位")
private String password;
```

## 2. 边界值分析法

### 核心思想
错误常发生在边界值附近，重点测试边界及其相邻值。

### 边界值选择技巧
```
假设范围 [min, max]：
  - min-1（略小于最小值）
  - min（最小值）
  - min+1（略大于最小值）
  - 典型值（中间任意值）
  - max-1（略小于最大值）
  - max（最大值）
  - max+1（略大于最大值）
```

### 实战示例：订单数量校验

```
需求：订单数量 1-999

边界值测试用例：
  - 数量 = 0（无效，低于最小值）
  - 数量 = 1（有效，最小边界）
  - 数量 = 2（有效，最小值+1）
  - 数量 = 500（有效，中间值）
  - 数量 = 998（有效，最大值-1）
  - 数量 = 999（有效，最大边界）
  - 数量 = 1000（无效，超过最大值）
  - 数量 = -1（无效，负数）
  - 数量 = 999999（无效，超大数）
```

### B端后台常见边界场景
| 场景 | 边界值 |
|------|--------|
| 分页查询 | 第1页、第0页(-1)、超过最大页、9999页 |
| 金额字段 | 0、0.01、999999.99、最大值、超大值 |
| 日期范围 | 最小日期、最大日期、跨年、跨月 |
| 文件上传 | 0KB、1KB、10MB(限制)、10.1MB(超限) |

## 3. 场景法（事件流分析法）

### 核心思想
通过描述用户操作路径和系统响应，构造场景进行测试。

### 场景分类
```
基本流（Happy Path）：用户正确、完整地完成操作
备选流1：用户输入错误，系统提示并允许重试
备选流2：网络中断，系统处理异常
备选流3：权限不足，无法完成操作
...
```

### 实战示例：采购审批流程

```
场景1：正常审批流程（基本流）
  1. 用户A提交采购申请
  2. 部门经理B审核通过
  3. 财务C复核通过
  4. 采购完成，状态更新为"已完成"

场景2：申请被驳回
  1. 用户A提交采购申请
  2. 部门经理B驳回申请（原因：预算不足）
  3. 用户A修改申请后重新提交

场景3：审批超时自动通过
  1. 用户A提交采购申请
  2. 部门经理B超过72小时未处理
  3. 系统自动通过审批

场景4：财务复核拒绝
  1. 用户A提交采购申请
  2. 部门经理B审核通过
  3. 财务C复核拒绝（原因：发票不规范）

场景5：并发审批（同一申请被多人同时处理）
  1. 用户A提交采购申请
  2. 部门经理B和部门经理C同时看到申请
  3. B先点击"通过"，C后点击"通过"
  4. 系统应处理并发，只执行第一个操作
```

## 4. 正交试验法

### 核心思想
使用正交表选取少量代表性测试组合，覆盖大多数因素组合。

### 适用场景
- 多个输入参数组合测试
- 配置项组合测试
- 兼容性测试矩阵

### 实战示例：用户搜索筛选条件

```
因素和水平：
  性别：全部、男、女
  状态：启用、禁用、全部
  部门：技术部、销售部、管理层、全部
  角色：管理员、普通用户、全部

使用正交表 L9(3^4)：
  测试用例1：全部、启用、技术部、管理员
  测试用例2：男、禁用、销售部、普通用户
  测试用例3：女、全部、管理层、管理员
  测试用例4：全部、女、普通用户、全部
  测试用例5：男、全部、技术部、普通用户
  测试用例6：女、启用、管理层、全部
  测试用例7：全部、男、管理层、普通用户
  测试用例8：男、女、技术部、全部
  测试用例9：女、启用、销售部、管理员
```

### 常用正交表工具
- 使用 AllPairs 工具生成最小测试集
- 或使用在线正交表生成器

## 5. 判定表法

### 适用场景
多个条件组合，每种组合对应不同结果的场景。

### 实战示例：订单优惠计算

```
条件：
  C1：订单金额 >= 1000
  C2：用户是VIP
  C3：使用优惠券

结果：
  A1：原价
  A2：95折
  A3：9折
  A4：85折

判定表：
| 条件组合 | C1=Y,C2=Y,C3=Y | C1=Y,C2=Y,C3=N | C1=Y,C2=N,C3=Y | C1=N,C2=Y,C3=Y |
|----------|----------------|----------------|----------------|----------------|
| 结果     | 85折           | 9折            | 95折           | 95折           |
```

## 6. 错误推测法

### 核心思想
基于经验推测可能存在的错误，重点测试。

### B端后台常见错误场景

```
1. 空值相关
   - 必填字段为空提交
   - 搜索条件为空查询
   - 批量操作包含空ID

2. 重复操作
   - 重复提交表单（双击按钮）
   - 重复审核同一单据
   - 重复删除同一数据

3. 边界条件
   - 分页到最后一页再下一页
   - 输入超长字符串
   - 日期范围选择错误（结束日期早于开始日期）

4. 并发场景
   - 多人同时编辑同一记录
   - 库存不足时多人同时下单
   - 审批流并发处理

5. 权限相关
   - 无权限用户访问URL直接进入
   - 角色切换后数据未刷新
   - 子账号权限不足时操作
```

## 7. 综合应用策略

### 测试用例设计流程

```
1. 需求分析
   ↓
2. 划分功能模块
   ↓
3. 确定输入和输出
   ↓
4. 应用等价类划分
   ↓
5. 应用边界值分析
   ↓
6. 应用场景法（补充异常流程）
   ↓
7. 使用正交法（多因素组合）
   ↓
8. 错误推测（补充经验场景）
   ↓
9. 评审测试用例
```

### B端管理后台测试覆盖建议

| 功能类型 | 主要方法 | 测试点 |
|----------|----------|--------|
| 表单录入 | 等价类+边界值 | 字段校验、必填、格式 |
| 列表查询 | 边界值+正交法 | 分页、筛选、排序 |
| 审批流程 | 场景法 | 多路径、异常流、并发 |
| 数据导出 | 边界值+错误推测 | 大数据量、格式 |
| 批量操作 | 错误推测 | 部分失败、全部失败 |
