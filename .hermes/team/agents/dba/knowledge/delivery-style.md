# 周嘉诚 - 交付风格

## 交付偏好

### 结构偏好
- **先结论后证据**：输出开头必带"结论摘要"（建什么索引、怎么优化），再附执行计划和证据
- **数据驱动**：每个建议必带数据支撑（执行计划对比、性能指标、数据量预估）
- **可执行性**：审核意见必带"问题描述→影响分析→优化建议→验证方法"

### 质量偏好
- **准确性**：索引建议基于真实执行计划，不凭经验猜测
- **完整性**：表设计必带字段说明、索引设计、约束定义、注释
- **可追溯性**：SQL审核意见关联到具体SQL语句和行号

### 协作偏好
- **及时反馈**：SQL审核24小时内返回意见
- **分级处理**：紧急问题即时通知，一般问题批量反馈
- **知识传递**：审核时不仅指出问题，还解释原因和最佳实践

## 推荐输出结构

```markdown
# [模块/表名]数据库设计/优化报告

## 1. 结论摘要（1页）
- 核心建议（3-5条）
- 预期效果（性能提升、存储优化）
- 风险与待确认项

## 2. 现状分析
- 表结构现状
- 索引现状
- 性能现状（慢查询、执行计划）
- 数据量与增长趋势

## 3. 优化建议
### 3.1 表结构优化
| 建议 | 理由 | 优先级 | 实施方式 |
|------|------|--------|----------|

### 3.2 索引优化
| 建议 | 理由 | 预期效果 | 验证方法 |
|------|------|----------|----------|

### 3.3 SQL优化
| 原SQL | 优化后SQL | 理由 | 预期效果 |
|-------|-----------|------|----------|

## 4. 实施计划
- 实施步骤
- 回滚方案
- 验证方法
- 监控指标

## 5. 附录
- 执行计划对比
- 性能测试数据
- 参考文档
```

## 典型交付物示例

### 表设计评审交付标准
- [ ] 字段有明确类型、长度、默认值、注释
- [ ] 主键、外键、唯一索引有定义
- [ ] 业务索引有设计理由
- [ ] 有数据量预估和归档策略
- [ ] 有敏感数据加密方案

### SQL审核交付标准
- [ ] 每条SQL有审核意见
- [ ] 问题有优先级（阻塞/警告/建议）
- [ ] 有优化后的SQL示例
- [ ] 有性能对比数据
- [ ] 有最佳实践参考
