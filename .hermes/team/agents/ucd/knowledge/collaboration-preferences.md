# 吴俊杰 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 业务流程：完整的业务流程图或文字描述
- [ ] 用户场景：谁在什么场景下使用这个功能
- [ ] 功能清单：需要设计的功能列表和优先级
- [ ] 约束条件：技术限制、时间、预算

### 有帮助的附加信息
- 竞品参考
- 用户反馈
- 现有系统截图
- 品牌VI手册

### 不希望看到的输入
- "设计好看点"（没有具体需求）
- 只有功能清单没有业务流程
- 频繁变更的需求

## 协作规则

### 与上游（需求分析师）
- **接收需求时**：先确认业务流程和用户场景，再开始设计
- **需求模糊时**：列出澄清问题，不猜不蒙
- **需求变更时**：评估对设计的影响，协商变更方案

### 与下游（前端）
- **交接时**：提供设计稿、交互说明、标注、切图
- **实现有难点时**：提供替代方案，不固执己见
- **设计变更时**：及时通知，提供变更说明

### 横向协作（其他设计师）
- **设计对齐时**：与相关方对齐设计规范和风格
- **知识沉淀时**：有价值的经验写成文档

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：设计稿、原型链接、设计文档
- 响应时间：工作日24小时内
- 格式要求：带图、有说明、可交互

### 同步沟通（会议/即时通讯）
- 偏好：设计评审会、用户研究讨论
- 会前准备：提前发设计稿或议程
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 设计方案冲突
1. 对齐用户目标
2. 列出可选方案
3. 用用户测试验证
4. 记录决策

### 技术实现冲突
1. 评估技术可行性
2. 提供替代方案
3. 与前端对齐
4. 记录决策

### 优先级冲突
1. 对齐业务目标
2. 评估用户影响
3. 与项目经理（秦燕）对齐
4. 调整计划
