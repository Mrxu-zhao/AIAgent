# 仓库级控制平面设计文档

## 1. 设计背景

本设计面向 `d:\KIMIK2.5\AIAgent` 全仓库，目标是在不先重写现有调度框架的前提下，建立一套可落地的“仓库级控制平面”，用于承接评估报告中提出的系统性优化工作。控制平面不直接替代业务逻辑与底层模块，而是作为统一的任务治理层，负责任务拆分、依赖编排、状态同步、冲突检测、并行执行协调、失败隔离、结果聚合与端到端验证。

本设计的重点不是单点修复某一个模块，而是建立一套适用于全仓库演进的执行机制，使 `.hermes/team/调度框架` 的优化、文档治理、协议收敛、性能验证等工作可以被多个智能体并发执行，同时保持过程可追踪、结果可复验、失败可隔离。

## 2. 设计目标

### 2.1 核心目标

- 将评估报告中的优化建议拆分为多个独立、可并行执行的子任务。
- 为每个子任务定义唯一标识、优先级、输入输出契约、依赖关系与验收标准。
- 建立统一的通信协议、状态同步机制和冲突检测规则。
- 通过超时、重试与回滚策略，保证单一子任务失败不阻断无关工作流。
- 在全部任务完成后自动聚合结果，并执行端到端回归与性能对比。
- 产出可维护、可审计、可扩展的长期治理文档。

### 2.2 非目标

- 本阶段不直接把现有调度框架改造成完整分布式平台。
- 本阶段不要求一次性解决评估报告中的全部 P2 长期治理项。
- 本阶段不以“全仓库所有功能统一提速”为主目标，性能验证以调度框架专项指标为主口径。

## 3. 备选方案

### 方案 A：轻量编排层

在现有仓库上方增加最小任务登记、状态文件与日志目录，仅协调并行执行，不改变底层结构。

优点：

- 上手快，侵入性低。
- 适合快速建立第一版并行治理机制。

缺点：

- 底层能力割裂依旧明显。
- 状态一致性、冲突检测和回归自动化能力偏弱。

### 方案 B：仓库级控制平面

在全仓库之上增加统一任务协议、状态仓、事件流、冲突检测、聚合器与回归验证层，将调度框架优化作为其中一个并行 workstream。

优点：

- 最符合“多智能体并发执行 + 状态共享 + 统一聚合”的目标。
- 能在不重写底层平台的情况下先建立可用治理层。
- 后续可平滑演进到更强的控制平面。

缺点：

- 前期设计要求高，必须先明确任务模型与运行规则。

### 方案 C：先平台化后执行

先重构调度框架本身为可靠平台，再由它去驱动全仓库优化。

优点：

- 长期架构上更理想。

缺点：

- 前置成本高，短期无法快速开始并行优化。
- 会把“先修平台”变成单点瓶颈。

### 方案选择

本设计选择 **方案 B：仓库级控制平面**。原因是它在工程投入、阶段收益与可落地性之间取得了最好平衡。

## 4. 总体架构

### 4.1 分层结构

控制平面总体分为五层：

1. **控制入口层**
   - 负责读取优化计划、注册任务、解析依赖、分配 workstream、监控执行生命周期。
2. **统一状态与通信层**
   - 负责任务协议、状态仓、事件流、冲突检测、任务锁。
3. **并行 Workstream 层**
   - 各智能体在不同主线上并发执行独立子任务。
4. **聚合与验证层**
   - 汇总任务结果，执行端到端回归与性能对比。
5. **故障隔离与回滚层**
   - 处理任务失败、依赖阻塞、局部回滚与恢复。

### 4.2 推荐的第一批 Workstream

- `WS-A`：调度框架正确性修复
- `WS-B`：通信协议与状态治理
- `WS-C`：性能基线、回归与对比验证
- `WS-D`：文档与可维护性治理

## 5. 子任务模型

### 5.1 Task Card 设计

每个子任务必须被建模为一个独立 `Task Card`，且只解决一个清晰问题。单个任务不允许同时承担代码修复、协议设计、压测和文档收口等多类职责。

### 5.2 唯一标识规则

任务 ID 采用如下格式：

```text
<Workstream>-<Priority>-<Sequence>
```

示例：

- `WS-A-P0-001`
- `WS-B-P1-002`
- `WS-C-P1-001`

规则：

- `WS-*` 表示所属主线
- `P0/P1/P2` 表示优先级
- `Sequence` 为三位流水号
- 任务 ID 一经分配不可复用，不因重试改变

### 5.3 Task Card 字段

每个任务卡必须至少包含：

- `task_id`
- `title`
- `goal`
- `scope`
- `lock_scope`
- `inputs`
- `outputs`
- `dependencies`
- `owner_agent`
- `review_agent`
- `priority`
- `timeout`
- `retry_policy`
- `rollback_policy`
- `acceptance_criteria`
- `status`
- `evidence`

### 5.4 输入 / 输出契约

输入必须显式声明：

- 来源文档或代码路径
- 前置任务输出
- 可使用的基线数据
- 环境与约束条件

输出必须至少包含：

- 结果产物路径
- 执行摘要
- 验证结果
- 风险与遗留问题

### 5.5 依赖表达

依赖分为三类：

- `hard_dependency`
- `soft_dependency`
- `shared_baseline`

其中：

- `hard_dependency` 未满足时任务不可启动
- `soft_dependency` 未完成不阻断执行，但允许后续二次修正
- `shared_baseline` 共享同一基线数据，但彼此不阻塞

### 5.6 第一批推荐任务

#### WS-A 调度框架正确性

- `WS-A-P0-001`：修复 `Monitor.get_dashboard_data()` 死锁
- `WS-A-P0-002`：修复 `WorkflowEngine` 忽略 `step.agent`
- `WS-A-P0-003`：补齐 `monitor.start()` 生命周期管理
- `WS-A-P0-004`：建立最小回归测试

#### WS-B 协议与治理

- `WS-B-P1-001`：定义统一任务协议
- `WS-B-P1-002`：定义状态仓结构与状态机
- `WS-B-P1-003`：定义冲突检测规则与锁域
- `WS-B-P1-004`：定义结果聚合格式

#### WS-C 性能与验证

- `WS-C-P1-001`：建立性能基线采集脚本
- `WS-C-P1-002`：建立端到端回归测试清单
- `WS-C-P1-003`：建立前后对比报告模板

#### WS-D 文档与维护性

- `WS-D-P1-001`：修正文档与实现不一致
- `WS-D-P1-002`：生成执行日志与维护说明模板
- `WS-D-P2-001`：沉淀长期演进手册

## 6. 通信协议与状态同步

### 6.1 核心对象

统一运行时必须围绕以下三类对象工作：

- `Task Card`：任务定义
- `Task Event`：任务生命周期事件
- `Task Artifact`：任务产物索引

自然语言说明可以存在，但不允许替代结构化事件与状态记录。

### 6.2 事件类型

建议固定以下事件类型：

- `task_registered`
- `task_ready`
- `task_started`
- `task_progress`
- `task_blocked`
- `task_conflict_detected`
- `task_retrying`
- `task_failed`
- `task_rolled_back`
- `task_completed`
- `artifact_published`
- `review_passed`
- `review_failed`

### 6.3 事件字段

每条事件至少包含：

- `event_id`
- `task_id`
- `event_type`
- `agent_id`
- `timestamp`
- `attempt`
- `status_before`
- `status_after`
- `summary`
- `artifact_refs`
- `lock_scope`
- `depends_on`
- `metrics_delta`
- `error_code`

### 6.4 状态机

统一状态定义为：

- `planned`
- `ready`
- `running`
- `blocked`
- `reviewing`
- `done`
- `failed`
- `rolled_back`
- `cancelled`

关键规则：

- 任务不得跳过 `reviewing` 直接进入 `done`
- `done` 必须伴随 `artifact_published` 与 `review_passed`
- `rolled_back` 表示已恢复到安全状态，而不是简单失败

### 6.5 状态同步模式

建议采用 **状态仓 + 事件追加日志** 的双轨模型。

- 状态仓保存当前快照
- 事件日志保存完整历史

快照至少应包含：

- `version`
- `last_event_id`
- `updated_at`
- `updated_by`

更新原则：

- 先写事件，再更新快照
- 快照版本落后时必须拒绝覆盖并要求重新拉取
- 聚合器只接受“最新快照 + 全量事件”的组合

## 7. 冲突检测、超时、重试与回滚

### 7.1 冲突检测

建议定义三层冲突：

- `file_lock_scope`
- `module_lock_scope`
- `contract_lock_scope`

处理原则：

- 文件级冲突：硬阻断并发执行
- 模块级冲突：软阻断或进入监视状态
- 契约级冲突：自动触发 review 或依赖重评估

### 7.2 超时策略

建议默认超时：

- `P0 代码修复类`：30-45 分钟
- `协议 / 文档类`：20-30 分钟
- `性能压测类`：45-90 分钟
- `聚合 / 回归类`：30-60 分钟

超时后动作：

- 发出 `task_failed` 或 `task_blocked`
- 保留当前产物与日志快照
- 由重试策略决定是否继续

### 7.3 重试策略

失败分为三类：

- `transient`
- `deterministic`
- `dependency`

建议：

- `transient`：自动重试 1-2 次
- `deterministic`：默认不自动重试
- `dependency`：转为 `blocked`

### 7.4 回滚策略

回滚分为：

- `code rollback`
- `state rollback`
- `artifact rollback`

规则：

- 仅作用于当前任务锁域
- 必须记录 `rollback_reason`
- 必须产出 `rollback_report`
- 不能抹除失败证据

### 7.5 失败隔离原则

失败传播只沿依赖图传播：

- 当前失败任务 -> `failed`
- 直接依赖它的任务 -> `blocked`
- 无依赖关系的任务继续执行

## 8. 端到端回归与性能验证

### 8.1 验证层次

建议分为两层：

- `专项验证`：以调度框架为主
- `全局交付验证`：验证仓库级控制平面本身

主口径以调度框架专项验证为主。

### 8.2 基线采集

在执行优化前必须完成一次 `Baseline Run`。

建议采集以下四类数据：

- 任务分发延迟
- 工作流执行延迟
- 监控接口响应时间
- CPU 资源占用

采样要求：

- 每项测试运行 10-30 次
- 记录 `avg / p95 / max`
- CPU 记录平均值与峰值
- 所有数据必须带时间戳与代码版本

### 8.3 回归场景

至少定义以下端到端场景：

- `S1 单任务分发`
- `S2 指定 agent 分发`
- `S3 标准工作流执行`
- `S4 并行子任务执行`
- `S5 冲突检测与失败隔离`
- `S6 聚合与回归报告生成`

### 8.4 核心性能指标

建议指标：

- `dispatch_latency_ms`
- `workflow_step_latency_ms`
- `dashboard_response_ms`
- `e2e_workflow_duration_ms`
- `cpu_avg_percent`
- `cpu_peak_percent`
- `cpu_per_task_percent`

### 8.5 达标规则

强制门槛：

- 所有 P0 回归场景通过
- 不新增死锁、崩溃、状态错乱
- 指定 agent 语义正确
- 聚合器可输出完整报告

性能目标：

- 延迟主指标下降 `>= 20%`
- CPU 平均利用率下降 `>= 15%`

稳定性要求：

- 多轮运行结果波动在可接受范围内
- 重试与回滚不破坏状态一致性

## 9. 最终交付物

控制平面完成后应自动输出以下四类核心产物：

- `任务拆分清单`
- `并行执行日志`
- `性能对比报告`
- `可维护性说明`

最终完整文档建议采用如下结构：

1. 执行概览
2. 任务拆分清单
3. 并行执行过程日志
4. 冲突与失败处理记录
5. 回滚与重试记录
6. 端到端回归测试结果
7. 性能对比报告
8. 目标达成情况
9. 遗留问题与风险
10. 后续可维护性说明

## 10. 实施边界与推荐顺序

推荐顺序如下：

1. 先完成 `WS-B` 的协议与状态治理底座
2. 并行推进 `WS-A` 正确性修复与 `WS-D` 文档治理
3. 在协议底座稳定后启动 `WS-C` 基线采集与回归验证
4. 所有子任务完成后，由聚合器统一生成结果文档并验收

该顺序能够确保在真正开始多智能体并发执行前，先有最小可用的状态同步与冲突控制能力。

## 11. 风险与约束

- 仓库内已有未提交改动，实施时必须严格控制提交范围。
- 当前调度框架仍是单进程内存原型，控制平面应避免把它误当成强一致运行时。
- 如果协议层先缺失就直接并发执行，极易引入结果漂移、状态混乱和提交冲突。
- 性能目标必须建立在同口径基线上，禁止直接用不同场景数据做前后对比。

## 12. 结论

仓库级控制平面的价值不在于替代所有现有模块，而在于为全仓库的并行治理建立统一约束。只要任务卡、事件流、状态仓、冲突检测、失败隔离和回归验证这六个基础能力建立起来，多智能体协作就能够从“临时协同”升级为“可治理的并行执行系统”。

本设计建议先以调度框架专项优化为主战场建立第一版闭环，再将其逐步推广到更广的仓库治理场景。
