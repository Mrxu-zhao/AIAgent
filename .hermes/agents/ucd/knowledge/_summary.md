# 经验教训汇总

> 从每个项目中提取的经验教训

## 2026-04-29 首次项目分析实战

### 经验1: 子Agent任务分配后的自我进化机制缺失

- **事件**: 4个子Agent并行分析后，仅生成了项目分报告，但没有按技能规范进行自我进化
- **原因**: 子Agent只完成了"任务执行"步骤，跳过了"归档进化"步骤
- **教训**: 主Agent在分配任务时应明确要求子Agent**同时**完成知识库进化，而不只是输出报告
- **改进**: 下次分派任务时，在context中明确要求：
  ```
  输出报告到 projects/ 后，同步完成知识库进化：
  1. 将经验写入 lessons_learned/
  2. 将模式写入 patterns/
  3. 更新 status.md
  ```

### 经验2: 微服务业务边界划分的重要性

- **事件**: xzmeto 后端分析发现 Flowable 工作流作为独立服务（xzmetro-flow），但实际业务表却在 xzmetro-app-server 中
- **教训**: 微服务拆分时，业务边界要清晰。Flowable 属于基础设施服务，其表应该在 flow 模块自己管理，不应混入业务服务
- **应用**: 分析微服务架构时，要同时关注"代码模块"和"数据库表"的归属是否一致

### 经验3: 前端多端分析时要注意技术栈差异

- **事件**: mall4j 四套前端中，PC端用 Vue3+Element Plus，移动端用 uni-app+sard-uniapp，差异很大
- **教训**: 分析多端配套项目时，先确认各端技术栈是否一致，再决定分析策略
- **应用**: 技术栈一致的可以横向对比分析，技术栈差异大的要分开深入分析
