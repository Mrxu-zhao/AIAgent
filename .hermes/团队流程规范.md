# StudyHelper 团队流程规范 v3.0

## 一、定位

本规范用于定义 StudyHelper 团队的统一协作基线，目标是把团队约定转换为可执行的 workflow 规则，而不是停留在文档描述层。

本文件只定义团队级通用规范，不把具体项目流程写死在代码里。具体流程实例必须存放在 `.hermes/workflows/` 目录下，以显式 workflow 文件的方式维护。

## 三、团队级强制阶段

所有交付类流程都必须从以下阶段中选择并显式建模，不允许再以静态 TASKS 绕过：

1. 需求确认
2. 需求分析
3. 需求评审
4. UCD设计
5. UCD评审
6. 架构设计
7. 架构评审
8. 数据库设计
9. 数据库评审
10. 前后端开发
11. 单元测试
12. 功能测试
13. 缺陷修复
14. 回归测试
15. 闭环确认
16. 发布审批
17. 部署上线

说明：

- 并非每个流程都必须覆盖以上全部阶段，但缺省流程必须给出是否启用该阶段的显式定义。
- 需求确认、各类评审、测试闭环、闭环确认、发布审批属于高优先级治理节点。
- 任何省略都必须在 workflow 文件中有明确理由，不能依靠隐式跳过。

## 四、UCD 设计规范

UCD 设计属于正式流程阶段，不能再作为前端开发的隐含前置条件。

### 4.1 UCD 设计输入

- 需求确认单
- PRD
- 目标用户和核心场景说明
- 信息架构或业务流程说明

### 4.2 UCD 设计输出

- 页面信息架构
- 关键流程原型
- 交互说明
- 状态与异常反馈说明
- 设计评审结论

### 4.3 UCD 设计准入

- 未完成需求评审，不允许进入 UCD 设计
- 未输出原型或交互说明，不允许进入 UCD 评审
- UCD 评审未通过，不允许进入前端开发

### 4.4 UCD 评审关注点

- 用户路径是否完整
- 关键信息层级是否清晰
- 交互反馈是否完整
- 异常态、空态、加载态是否定义
- 是否与 PRD、架构设计约束一致

## 五、workflow 映射规范

团队流程规范必须映射成显式 workflow 定义，统一放在 `.hermes/workflows/`。

在 Hermes 执行模式下，项目经理角色由 Hermes agent 以虚拟治理角色 `project-manager` 承担。workflow 中凡是项目经理审批节点，应显式声明 `agent: "project-manager"`，并保留 `approval_role: "项目经理"` 作为责任角色标识。

每个 workflow 文件至少包含：

- `workflow_id`
- `name`
- `description`
- `variables`
- `steps`

每个 `step` 至少包含：

- `id`
- `name`
- `type`
- `agent`
- `task`
- `dependencies`

### 5.1 step 准入字段

为了把团队规范落成可执行校验，每个 step 或 task card 可声明以下准入字段：

- `required_deliverables`
- `coverage_threshold`
- `test_pass_rate`
- `defect_closure_rate`
- `approval_required`
- `approval_role`

### 5.2 字段含义

| 字段 | 含义 | 示例 |
|------|------|------|
| `required_deliverables` | 执行前必须已具备的交付物 | `["PRD.md", "UCD设计稿.md"]` |
| `coverage_threshold` | 覆盖率阈值，支持前后端分开定义 | `{"frontend": 60, "backend": 70}` |
| `test_pass_rate` | 测试通过率阈值 | `100` |
| `defect_closure_rate` | 缺陷关闭率阈值 | `{"P0": 100, "P1": 100, "P2": 90}` |
| `approval_required` | 是否需要阻塞审批 | `true` |
| `approval_role` | 负责审批的角色 | `"项目经理"` |

### 5.3 阻塞规则

- `approval_required=true` 时，必须等待真实审批结果，不能返回占位成功。
- 任一准入字段不满足时，step 状态必须进入阻塞或失败，不能继续执行后续步骤。
- `control-plane-run` 和 `workflow` 必须走同一套 workflow 执行入口，禁止直接运行静态 TASKS 绕过流程。

## 六、默认项目开发流程

默认项目开发流程应至少包含以下顺序与依赖关系：

```text
需求确认
  -> 需求分析
  -> 需求评审
  -> UCD设计
  -> UCD评审
  -> 架构设计
  -> 架构评审
  -> 数据库设计
  -> 数据库评审
  -> 前后端开发
  -> 单元测试
  -> 功能测试
  -> 缺陷修复
  -> 回归测试
  -> 闭环确认
  -> 发布审批
  -> 部署上线
```

其中以下节点属于强阻塞节点：

- 需求确认
- 需求评审
- UCD评审
- 架构评审
- 数据库评审
- 闭环确认
- 发布审批

## 七、测试闭环规范

### 7.1 单元测试

- 开发完成代码的同时必须完成单元测试
- 未编写单元测试，不允许提测
- 前端覆盖率最低要求 `>= 60%`
- 后端覆盖率最低要求 `>= 70%`
- 单元测试通过率必须 `= 100%`

### 7.2 功能测试

- 功能测试执行率必须 `= 100%`
- 冒烟测试通过率必须 `= 100%`
- 功能测试通过率默认要求 `>= 95%`

### 7.3 缺陷修复

- P0 缺陷关闭率必须 `= 100%`
- P1 缺陷关闭率必须 `= 100%`
- P2 缺陷关闭率必须 `>= 90%`

### 7.4 回归测试

- 所有已修复缺陷必须经过回归验证
- 回归测试需验证修复未引入新问题
- 回归测试报告必须纳入闭环确认输入

### 7.5 闭环确认

闭环确认前必须具备以下交付物：

- 单元测试报告
- 功能测试报告
- 缺陷报告
- 缺陷关闭报告
- 回归测试报告
- 发布检查清单

## 九、新增流程规则

如需新增流程，不允许直接改写默认流程逻辑，必须：

1. 在 `.hermes/workflows/` 下新增独立流程文件
2. 继承本规范中的强制准入和审批原则
3. 显式声明启用和跳过的阶段
4. 保证统一通过 workflow 入口执行

示例：

- `project_delivery.json`: 标准项目开发流程
- `hotfix_release.json`: 紧急修复流程
- `ucd_only.json`: 纯设计交付流程
- `data_migration.json`: 数据迁移流程
