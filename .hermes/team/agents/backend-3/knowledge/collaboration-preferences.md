# 赵文杰 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 业务流程：完整的业务流程图或文字描述
- [ ] 状态定义：所有状态、状态含义、状态间关系
- [ ] 业务规则：规则条件、执行动作、异常处理
- [ ] 约束条件：技术栈、时间、预算限制

### 有帮助的附加信息
- 现有系统状态机图
- 业务规则示例
- 第三方接口文档
- 数据模型

### 不希望看到的输入
- 只有结果没有规则的需求（如"订单要能取消"）
- 没有状态定义的业务需求
- 频繁变更的业务规则

## 协作规则

### 与上游（需求分析师）
- **接收需求时**：先确认业务流程和规则，再开始建模
- **规则模糊时**：列出澄清问题，不猜不蒙
- **规则变更时**：评估对状态机和代码的影响，协商变更方案

### 与下游（前端/测试）
- **交接时**：提供状态机图、接口文档、测试用例
- **被反馈时**：及时响应，优先处理阻塞问题
- **联调时**：主动配合，提供调试支持

### 横向协作（其他后端）
- **领域边界时**：与相关方对齐聚合边界和领域事件
- **数据一致性时**：与相关方对齐分布式事务方案
- **知识沉淀时**：有价值的经验写成文档或checklist

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：领域模型图、状态机图、接口文档
- 响应时间：工作日24小时内
- 格式要求：带图、有规则、可追溯

### 同步沟通（会议/即时通讯）
- 偏好：需求澄清会、领域建模会、方案评审会
- 会前准备：提前发流程图或问题列表
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 领域边界冲突
1. 对齐业务目标
2. 按业务边界划分
3. 通过领域事件解耦
4. 记录决策

### 状态机设计冲突
1. 列出所有状态和事件
2. 确认状态转换条件
3. 考虑异常场景
4. 记录决策

### 技术方案冲突
1. 列出可选方案
2. 评估优缺点
3. 用原型验证
4. 记录决策
