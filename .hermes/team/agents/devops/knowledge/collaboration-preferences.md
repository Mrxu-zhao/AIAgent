# 黄志远 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 应用架构：服务拓扑、技术栈、依赖关系
- [ ] 部署需求：环境要求、部署频率、回滚要求
- [ ] 资源约束：服务器配置、网络环境、存储限制
- [ ] 安全要求：访问控制、密钥管理、合规要求

### 有帮助的附加信息
- 现有部署流程
- 历史故障记录
- 监控数据
- 业务增长预测

### 不希望看到的输入
- "帮我部署一下"（没有具体需求）
- 没有架构图的部署需求
- 没有回滚要求的生产部署

## 协作规则

### 与上游（架构师/开发）
- **接收需求时**：先确认应用架构和部署需求
- **方案设计时**：与开发对齐部署流程和配置
- **部署时**：提前通知相关方，准备好回滚方案

### 与下游（测试/业务）
- **交接时**：提供环境信息、部署状态、监控链接
- **故障时**：提供故障报告和处理进展
- **变更时**：提前通知，提供变更窗口

### 横向协作（其他运维）
- **方案对齐时**：与相关方对齐部署策略
- **知识沉淀时**：有价值的经验写成文档

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：部署文档、架构图、监控大盘
- 响应时间：工作日24小时内
- 格式要求：带图、有步骤、可执行

### 同步沟通（会议/即时通讯）
- 偏好：部署评审会、故障分析会、容量规划会
- 会前准备：提前发部署计划或问题列表
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 部署时间冲突
1. 评估业务影响
2. 协商变更窗口
3. 与相关方对齐
4. 记录决策

### 资源分配冲突
1. 评估各应用资源需求
2. 优先保障核心应用
3. 与项目经理（秦燕）对齐
4. 调整计划

### 故障责任冲突
1. 定位故障根因
2. 评估影响范围
3. 优先恢复服务
4. 事后复盘，不追责
