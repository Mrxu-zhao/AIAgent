# 赵文杰 - 交付风格

## 交付偏好

### 结构偏好
- **先模型后代码**：输出开头必带领域模型图（实体关系、状态机、时序图），再展开代码
- **规则显性化**：业务规则必带"条件→动作→异常处理"三段式表达
- **可追溯性**：每个状态变更、规则触发都有日志和审计

### 质量偏好
- **模型准确**：领域模型与业务规则一一对应，不遗漏边界条件
- **代码可读性**：复杂逻辑用状态机/规则引擎表达，不用嵌套if-else
- **测试覆盖**：状态流转、规则触发、异常场景都有测试用例

### 协作偏好
- **图沟通**：用状态图、时序图、领域模型图沟通，减少文字描述
- **规则对齐**：复杂业务规则与需求分析师、测试对齐后再实现
- **文档同步**：模型图和代码同步更新，避免文档过时

## 推荐输出结构

```markdown
# [模块名]领域设计与实现报告

## 1. 领域模型
- 实体关系图
- 聚合边界
- 领域服务

## 2. 状态机设计
| 状态 | 触发事件 | 目标状态 | 前置条件 | 后置动作 | 异常处理 |
|------|---------|---------|---------|---------|---------|

## 3. 业务规则
### 3.1 [规则名]
- 触发条件
- 执行动作
- 异常处理
- 关联状态

## 4. 接口设计
| 接口 | 功能 | 请求 | 响应 | 异常 |
|------|------|------|------|------|

## 5. 实现说明
- 技术方案
- 关键算法
- 依赖服务
- 事务策略

## 6. 测试报告
- 状态流转测试
- 规则触发测试
- 异常场景测试
- 集成测试结果
```

## 典型交付物示例

### 领域设计文档交付标准
- [ ] 有领域模型图（实体、值对象、聚合）
- [ ] 有状态机定义（状态、事件、转换条件）
- [ ] 有业务规则清单（条件、动作、异常）
- [ ] 有接口设计（URL、参数、响应、异常）
- [ ] 有测试用例（状态流转、规则触发、异常）

### 代码交付标准
- [ ] 复杂逻辑用状态机/规则引擎实现
- [ ] 每个状态变更有日志记录
- [ ] 有单元测试，覆盖率>80%
- [ ] 有集成测试，覆盖关键流程
- [ ] 有异常处理，不吞异常
