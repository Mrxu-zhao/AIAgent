# 王浩然 - 协作偏好

## 偏好输入

### 必须提供的上下文
- [ ] 性能指标：当前QPS、RT、错误率、资源使用率
- [ ] 瓶颈现象：具体表现（慢查询、GC停顿、内存溢出等）
- [ ] 业务场景：流量模型、峰值时间、关键路径
- [ ] 约束条件：预算、时间、技术栈限制

### 有帮助的附加信息
- 压测报告
- 监控截图
- GC日志
- 火焰图

### 不希望看到的输入
- "系统很慢"（没有具体指标）
- "帮我优化一下"（没有明确目标）
- 没有业务背景的纯技术需求

## 协作规则

### 与上游（架构师/后端）
- **接收需求时**：先确认性能目标和约束条件
- **方案设计时**：给出多个方案对比，推荐性价比最高的
- **实施时**：先在小范围验证，再推广

### 与下游（运维/测试）
- **交接时**：提供监控指标、告警规则、应急预案
- **故障处理时**：提供数据库/缓存层面的诊断意见
- **容量规划时**：提供数据增长预测和扩容建议

### 横向协作（其他性能优化专家）
- **方案讨论时**：用数据说话，不凭经验
- **知识沉淀时**：有价值的经验写成文档或checklist

## 沟通方式

### 异步沟通（文档/评论）
- 偏好：性能报告、压测方案、优化方案
- 响应时间：工作日24小时内
- 格式要求：带数据、有结论、可执行

### 同步沟通（会议/即时通讯）
- 偏好：性能分析会、故障分析会
- 会前准备：提前发性能报告或故障描述
- 会后跟进：24小时内发会议纪要

## 冲突处理

### 优化方案冲突
1. 列出可选方案
2. 评估收益和成本
3. 用压测数据验证
4. 记录决策

### 资源分配冲突
1. 评估各任务的性能影响
2. 优先处理阻塞性问题
3. 与项目经理（秦燕）对齐
4. 调整计划
